# Migration: v0.8.1

## Summary
- **Release class:** patch (semver compatible). The `v0.8.0..HEAD` arc carries 1 `fix:` commit (`98736ea`, omit empty append-only arrays from the reconciled profile) and 2 `docs:` housekeeping commits (`ab3bd5c`, `82d5461`) with **zero `!:` subjects and no `BREAKING CHANGE:` footers** → PATCH bump; not minor/major. Last tag = `v0.8.0`; intended version = `v0.8.1`.
- **Upgrade path:** binary self-update, then re-render the corpus. Fully automatic for the default consumer — the fix ships through the Go binary's profile reconcile marshal path; `vh-agent-harness update` applies it. No profile, ownership-class, capability, or command-surface change.
- **Risk:** low. The change is backward-compatible: non-empty append-only arrays (`modules: [core]`, etc.) still emit normally. Only absent/empty arrays now stay absent instead of being re-emitted as `[]` on every non-noop reconcile.

The central thing to understand about this release: **the profile reconciler no longer re-emits empty append-only arrays.** The four append-only fields — `modules`, `overlays`, `policy_packs`, `capabilities` — now carry `omitempty`, so an absent/empty array stays absent in the reconciled `vh-harness-profile.yml` after `update`. This closes the long-standing v0.2.0 watch-out #3 condition, where an armed reconcile re-added `modules: []` (an empty list) on every update, defeating the documented `modules:` deprecation: a consumer who removed an empty `modules:` had it silently re-added, and the re-added empty stub emitted no deprecation warning (the warning gate is silent-on-empty by design), leaving the migration both uncompletable and undetectable.

## What `update` handles for you
- Re-renders from the v0.8.1 binary, whose reconcile marshal path now omits empty append-only arrays. After `update`, an absent/empty `modules:` (and `overlays:`/`policy_packs:`/`capabilities:`) stays absent in the reconciled `vh-harness-profile.yml` instead of being re-added as `[]`.
- Re-renders the embedded examples template so `vh-agent-harness example` no longer advertises a deprecated `modules:` block.
- The `modules:` deprecation warning is unchanged: it still fires ONLY on a NON-EMPTY `modules:`. An empty/absent `modules:` emits no warning (by design) and, as of this release, also no longer leaves a `modules: []` stub behind.
- No profile, ownership-class, capability, agent, command-surface, or runtime change beyond the above. No schema-reconciliation change beyond the empty-array drop.
- Everything tagged `local_adopted` or inactive `overlay_extension` is preserved as before.

## Watch-outs
1. **Supersedes v0.2.0 watch-out #3 (resolved).** v0.2.0 noted that the Go `Modules` field had no `omitempty`, so an armed reconcile emitted `modules: []` (an empty list) rather than dropping the key. As of v0.8.1, the four append-only array fields (`modules`, `overlays`, `policy_packs`, `capabilities`) carry `omitempty`, so empty append-only arrays now drop from reconciled *merged output*; an absent/empty `modules:` stays absent after `vh-agent-harness update`, resolving the v0.2.0 watch-out #3 condition. The v0.2.0 #3 case no longer reproduces; the deprecation migration it blocked (remove an empty `modules:` and have it stay removed) is now completable. `v0.2.0.md` is left unchanged as a frozen point-in-time record — this note supersedes it.
2. **Non-empty arrays are unaffected.** A profile carrying `modules: [core]` (or a non-empty `overlays:`/`policy_packs:`/`capabilities:`) still emits normally and, for `modules:`, still triggers the deprecation warning when non-empty. The fix only changes the empty/absent case.
3. **The deprecation warning is still silent-on-empty.** This release does not add a warning for an empty `modules: []`: the warning fires only on a non-empty `modules:` (unchanged). An empty `modules:` is accepted on read (it remains a valid top-level key) and stays silent, but it is **not durable** — any subsequent non-noop reconcile (e.g. a `vh-agent-harness update` that changes managed files) re-marshals the profile and drops the empty key again, consistent with the `omitempty` change this release ships. There is no supported way to retain a permanent `modules: []` stub in reconciled output; that empty-array drop is the intended behavior.

## Rollback
A patch release is reversible by binary downgrade: reinstall the `v0.8.0` binary and `vh-agent-harness update` re-renders the `v0.8.0` corpus (the render is authoritative for `platform_managed` files). **Caveats:** rolling back reintroduces the reconcile behavior where an absent/empty `modules:` (and `overlays:`/`policy_packs:`/`capabilities:`) is re-emitted as `[]` on the next non-noop reconcile — i.e. the v0.2.0 watch-out #3 condition returns. No data migration ran, so there is nothing to undo beyond the binary reinstall and re-render. A `local_adopted` customization survives the rollback as before.
