# Migration: v0.10.0

## Summary
- **Release class:** minor (semver compatible). The `v0.9.0..HEAD` arc carries a `fix:` (rendered overlay skills whose source was removed were left live, unreported, and invisible to `update --dry-run`) and its companion `feat:` (a versioned per-file render manifest + a preserved-orphan report surface), plus two further additive `feat:` commits in the same arc — `8a66bca` `feat(overlay): overlay docs` (a NEW consumer CLI subcommand) and `8afbe94` `feat(release-readiness): G6 skill-pilot evidence gate` (a domain-free S2 hold-ID + evidence-record contract added to the skill-creator reference under `templates/core/`). All four are additive/non-breaking with **zero `!:` subjects and no `BREAKING CHANGE:` footers** → MINOR bump; not patch/major. Last tag = `v0.9.0`; intended version = `v0.10.0`.
- **Upgrade path:** binary self-update, then re-render the corpus. Fully automatic for the default consumer — the fix and the new report surface ship through the Go binary's seam-apply path; `vh-agent-harness update` activates them. No profile, ownership-class, capability, or runtime change for the default consumer. The arc also carries two ADDITIVE consumer-visible surfaces alongside the renderstate centerpiece: the `vh-agent-harness overlay docs <name>` subcommand (`8a66bca`) and a domain-free S2 stable hold-ID + evidence-record contract added to the skill-creator reference under `templates/core/` (`8afbe94`); `templates/core/` remains domain-free.
- **Risk:** low. The change is additive and report-only. It never deletes a rendered file, never mutates `lineage.yml`, and never retroactively adopts pre-existing `.opencode/skills/` directories. A consumer with no prior manifest is bootstrapped forward-looking (baseline from the current successful render) and sees no orphans on first run.

The central thing to understand about this release: **the seam now remembers what it rendered, so a removed overlay skill source no longer vanishes silently.** Previously, removing an overlay skill source (e.g. deleting `.vh-agent-harness/overlays/<pack>/skills/<name>/`) and re-running `vh-agent-harness update` left the previously-rendered `.opencode/skills/<name>/` live and untouched (the renderer's deliberate conservative non-deleting posture), with no record that the harness had ever produced it — invisible to both `update` and `update --dry-run`. v0.10.0 introduces a separate versioned manifest at `.vh-agent-harness/rendered-outputs.json` that records producer-aware per-FILE render provenance (producing pack + source-relative path + digest + successful render id). On each `update`, the seam loads the prior manifest, compares it against the current render, and surfaces a "Preserved orphan skill file(s)" notice — naming the orphaned FILE by its full destination path — for any previously-rendered overlay skill whose producing SOURCE is now MISSING. Pack deselection (source still present, just not selected under `overlays:`) is explicitly NOT flagged, and project-added `.opencode/skills/` directories are never recorded so they are never flagged. The notice is **report-only**: v0.10.0 does not delete, does not add `--prune`, and does not auto-prune. (Bug reported by TrueAI against v0.9.0.)

## How to migrate (automated)
```bash
vh-agent-harness self-update            # pull the new binary (v0.10.0)
vh-agent-harness version                # expect: 0.10.0 (<label>)
vh-agent-harness update --dry-run       # ownership-safe preview; surfaces any preserved orphan skill(s) read-only
vh-agent-harness update                 # applies platform_managed render; writes the render manifest; prints preserved-orphan notices
vh-agent-harness doctor                 # lint the result
```
No profile, ownership-class, capability, or runtime change is required for the default consumer. The new `overlay docs <name>` subcommand is additive and used on demand (no migration action); the S2 hold contract re-renders as a domain-free skill-creator reference. Source-builders rebuilding from this tag are unaffected (no Go toolchain bump in this arc).

## What `update` handles for you
- Writes the new `.vh-agent-harness/rendered-outputs.json` manifest after a non-dry-run apply — but only when no currently-rendered, manifest-tracked overlay-skill destination reports a failed live write (non-skill write failures do not gate; `substrate.Apply` return semantics are unchanged) — recording per-file provenance for overlay-rendered skills. On the first run after upgrading (no prior manifest) it establishes a forward-looking baseline from the current render — it does NOT scan and adopt pre-existing `.opencode/skills/` directories.
- Loads the prior manifest on every `update` and `update --dry-run`, compares it against the current render, and prints a "Preserved orphan skill file(s)" notice — naming the orphaned FILE by its full destination path — for each previously-rendered overlay skill whose producing source is now missing and whose destination is still present.
- Guards manifest provenance on write failure: if a tracked overlay-skill destination's live write FAILED during the apply, the manifest is NOT advanced — the prior manifest is left byte-for-byte intact and a warning is printed to stderr. The apply's return semantics are unchanged (apply still reports success for a live-write failure); only the manifest persist is skipped so provenance never claims a file was rendered that was not. This is the narrow fix that keeps the v1 report truthful; the broader "make apply/lineage provenance truthful under live-write failure" work is tracked separately.
- Leaves all rendered files untouched. v0.10.0 is report-only: there is no deletion, no `--prune` flag, and no `prune` command. The renderer's conservative non-deleting posture is unchanged.
- Does not touch `lineage.yml` — the manifest is a SEPARATE file. The S1-only lineage invariant is unchanged.
- Everything tagged `local_adopted` or inactive `overlay_extension` is preserved as before.

## Watch-outs
1. **Report-only — no deletion in v1.** A preserved-orphan notice names the orphaned FILE by its full destination path and tells you it is still on disk; it does NOT remove it. Removing the file is a manual operator step: delete the named FILE if you no longer want it, and remove the whole skill directory only after verifying EVERY file in it is orphaned (a mixed directory may still contain actively-rendered files). A future prune command may automate this. The v1 scope is overlay-rendered SKILLS only; commands, agents, arbitrary overlay units, and core removals are out of scope.
2. **Pack deselection is NOT an orphan.** A skill whose source still exists in its pack but whose pack is no longer listed under `overlays:` is NOT flagged — the producing source is present, just not selected. Deselection and source-removal are distinct conditions; only source-removal is reported. To stop seeing a notice, either restore the source or manually delete the named orphaned FILE (after which the next `update` retires the record); remove the whole skill directory only after verifying EVERY file in it is orphaned.
3. **Project-added `.opencode/skills/` directories are never flagged.** The manifest only records what the harness rendered. A skill directory a human or project created directly under `.opencode/skills/` has no manifest record and is never reported as an orphan.
4. **Forward-looking bootstrap — no retroactive adoption.** On the first run after upgrading (no prior manifest), no orphans are reported even if stale rendered overlay skills already exist on disk, because there is no historical proof the harness produced them. Orphan reporting begins only for skills rendered AFTER the manifest is established.
5. **Supersedes the v0.9.0 rollback note (line 59 (d)).** v0.9.0's rollback text claimed that re-rendering with the older binary "removes the `debugging-loop` and `tdd-loop` skills from `.opencode/skills/`". That is misleading: the renderer never deletes a rendered file when its source disappears — it preserves it (exactly the condition this release now makes visible). A binary downgrade re-renders `platform_managed` files authoritatively but does NOT remove skills whose source is absent from the older binary; it leaves them preserved on disk. `v0.9.0.md` is left unchanged as a frozen point-in-time record — this note supersedes that rollback line.
6. **The manifest is binary-written state, not a rendered corpus file.** `.vh-agent-harness/rendered-outputs.json` is written by the seam-apply path at update time, not by the corpus renderer, and carries no ownership class in `core_manifest.go`. It is safe to delete (the next `update` re-establishes a forward-looking baseline); it is safe to commit (it is deterministic for a given render).
7. **`.local.json` overrides are LOCAL and gitignored — they are not shared.** A `X.local.json` companion under `.opencode/repo-configs/` is matched by the new scoped wildcard `.opencode/repo-configs/*.local.json` and never committed. This is intentional (per-developer / per-environment overrides), but it means a local field — including `mode` — silently overrides BOTH the user-level and committed-project config for whoever holds the local file, and teammates who do NOT have the local file see the committed behavior. If the team's `vh-agent-harness doctor` effective-`mode` output disagrees between machines, check for a `.local.json` companion before assuming a committed-config problem. Removing (or renaming) the local file restores the committed-project-over-user precedence on the next read.

## Rollback
A minor release is reversible by binary downgrade: reinstall the `v0.9.0` binary and `vh-agent-harness update` re-renders the `v0.9.0` corpus (the render is authoritative for `platform_managed` files). **Caveats:** rolling back (a) removes the preserved-orphan detection and report surface — `update` and `update --dry-run` no longer print orphan notices; (b) stops writing `.vh-agent-harness/rendered-outputs.json` — the file already on disk is left in place and harmlessly ignored by the v0.9.0 binary (it is not read by older code); (c) does NOT remove any rendered files — consistent with this note's supersession of the v0.9.0 rollback line 59 (d): a re-render never deletes a skill whose source is absent, it preserves it. No data migration ran, so there is nothing to undo beyond the binary reinstall and re-render. A `local_adopted` customization survives the rollback as before.

## overlay docs command

New in this arc (`8a66bca`): `vh-agent-harness overlay docs <name>` prints an overlay pack's `README.md` to stdout — the discovery mechanism for pack configuration reference. The pack is resolved the SAME way a render resolves it (a project-local pack at `.vh-agent-harness/overlays/<name>/` is preferred, then the embedded overlays filesystem shipped inside the binary, via `overlay.OpenPackFor`), and the name is validated with `nameRe` (no path traversal). It works for ANY pack name: shipped packs like `auto-classifier-pilot` or `release`, or your own project-local packs.

Who uses it: an adopter who has selected (or is evaluating) a pack and needs its configuration reference, mode tables, and enablement steps WITHOUT rendering the pack or reading its plugin source code. Pack READMEs are deliberately excluded from the rendered `.opencode/` tree (`overlay.IsPackDocFile` — a pack README never pollutes `.opencode/README.md`), so this command is the canonical way to read them. It is surfaced in `guide` output (an auto-classifier config step when that overlay is active, plus a general pack-docs tip for all installed states) and in `doctor` remediation hints. Notably, it surfaces the embedded `auto-classifier-pilot` README (~1100 lines covering every config field) that was previously embedded but invisible to consumers. A pack with no `README.md` lists its present doc files and points at `doctor` for the unit inventory.

## S2 skill-lifecycle hold contract (release-readiness relevant)

New in this arc (`8afbe94`): the skill-creator reference `skill-lifecycle.md` (under `templates/core/.opencode/skills/skill-creator/references/`, rendered into every consumer's `.opencode/skills/skill-creator/references/`) gains a "Stable hold-ID + evidence-record contract (release-relevant state)" subsection under its S2 rule (overlay-pilot-then-promote). It pins a two-surface contract that any release-readiness gate can cross-check by a STABLE JOIN KEY — never narrative prose:

- **Canonical backlog row (the hold):** a tagged row carrying a stable hold ID of the form `s2-hold: S2-<skill>-001`; authoritative for "a strict S2 hold exists".
- **Evidence-packet slot (the verdict):** a slot joined to that SAME hold ID, carrying a `PENDING`|`SATISFIED` verdict; authoritative for "the pilot succeeded".

Lifecycle: on hold → create the tagged row + a `PENDING` slot; on pilot landing → add real provenance + set `SATISFIED`; on resolution → close the backlog row ONLY after the slot is `SATISFIED`.

What it means for release-pack adopters: the contract ships to ALL consumers as a domain-free (token-only) skill-creator reference, so `templates/core/` remains domain-free. Adopters who hold skills/designs under S2 and run a release-readiness flow now have a stable, enumerable join key (`s2-hold:` prefix + `PENDING`/`SATISFIED` verdict) to prove a held skill's pilot landed before promotion. The G6 release-readiness gate that CONSUMES this contract — blocking the readiness HANDOFF (`handoff_to_releaser`) while any S2 hold is `PENDING`, missing, disagreed, duplicated, or ambiguous — is dogfood-local to THIS repo in this release (it lives in the `harness-dogfood` overlay and its 1:1 `.opencode/agents/harness-release-readiness.md` mirror, NOT in the shipped `release` pack, which contributes the tag-driven `releaser`). It demonstrates the contract; consumers do not receive a blocking G6 gate from this arc.

## .local.json repo-config override convention

New in this arc: a reusable per-repo override convention lives under
`.opencode/repo-configs/`. A committed base file `X.json` is optionally
accompanied by a gitignored `X.local.json` shallow field-override. The
convention is "documented convention + per-plugin JS implementation" — there
is no Go runtime loader and no shared JS utility; the consuming plugin reads
its own files at opencode tool-call runtime, and Go's role is limited to
`doctor` lint + effective-semantics parity (see "Promotion gate" below).

### First consumer: auto-classifier-pilot (auto-tool-gate)

The first (and currently only) consumer is the `auto-classifier-pilot`
overlay's `auto-tool-gate` plugin. Its behavior config now reads from up to
THREE physical files per type:

- **plugin (behavior) config:** `auto-gate-config.json` (committed-project,
  optionally committed) + `auto-gate-config.local.json` (project-local,
  gitignored) + user-level `~/.config/vh-agent-harness/auto-gate-config.json`
- **llm (credentials) config:** `auto-gate-llm.json` (NEVER committed —
  secrets-adjacent, separately gitignored) + `auto-gate-llm.local.json`
  (project-local, gitignored) + user-level `auto-gate-llm.json`

### Precedence (4-level, project-local HIGHEST)

```
defaults  ←  user-level  ←  committed-project  ←  project-local
                                                       (.local.json wins per field)
```

Shallow per-field merge then the existing normalization pass:
`normalize({...defaults, ...user, ...committedProject, ...projectLocal})`.

### Backward compatibility

`.local.json` is OPTIONAL and absent by default. A repo with only the
single-file `auto-gate-config.json` (the legacy shape) behaves EXACTLY as
before — the local layer is consulted only when the file exists, and an absent
local file is valid and silent.

### Operational consequence

A field present in `auto-gate-config.local.json` — including `mode` —
overrides the SAME field in BOTH the user-level file and the committed-project
file. The authoritative behavior config has **8 fields** (`enabled`, `mode`,
`stubVerdict`, `promptFile`, `replyMode`, `onUncertain`, `harnessContext`,
`guides`); any subset may appear in the local companion (a partial local file
inherits the lower layers for the fields it omits). An explicit falsy value
(`false`, `""`) in the local file is a REAL override, not treated as absence.

### doctor behavior

`vh-agent-harness doctor` now discovers and shape-lints the local files as
part of the `auto-classifier` check:

- **absent** local file → valid and silent (legacy behavior).
- **present and valid** → accepted independently; included in effective-value
  resolution so the reported effective `mode`/LLM semantics match what the
  plugin actually uses.
- **present but invalid** (malformed JSON, bad enum, wrong type, etc.) → the
  check FAILs, and the failure detail identifies the LOCAL layer (label
  contains `project-local`).

Doctor does NOT replicate the plugin's per-file mtime cache; it recomputes the
effective semantics from the present files each run.

### gitignore rule

The repo `.gitignore` gains a scoped wildcard for the convention
local-companions:

```
.opencode/repo-configs/*.local.json
```

The secrets-adjacent `auto-gate-llm.json` ignore is KEPT separate and is NOT
covered by the wildcard — LLM credentials are a different concern from local
overrides, and the LLM base file is never committed while the behavior base
file is optionally committable.

### Promotion gate (no shared-util extraction in this arc)

This release does NOT extract a shared `.local.json`-loading utility. The
convention is per-plugin by design. A shared util will be promoted ONLY when a
SECOND INDEPENDENT plugin family needs ALL of: derive `X.local.json`; optional
absence; shallow merge + normalize; per-file mtime invalidation; and
level-labelled diagnostics. Until then each consumer implements the pattern
itself.
