# Migration: v0.10.0

## Summary
- **Release class:** minor (semver compatible). The `v0.9.0..HEAD` arc carries a `fix:` (rendered overlay skills whose source was removed were left live, unreported, and invisible to `update --dry-run`) and its companion `feat:` (a versioned per-file render manifest + a preserved-orphan report surface) with **zero `!:` subjects and no `BREAKING CHANGE:` footers** → MINOR bump; not patch/major. Last tag = `v0.9.0`; intended version = `v0.10.0`.
- **Upgrade path:** binary self-update, then re-render the corpus. Fully automatic for the default consumer — the fix and the new report surface ship through the Go binary's seam-apply path; `vh-agent-harness update` activates them. No profile, ownership-class, capability, or command-surface change.
- **Risk:** low. The change is additive and report-only. It never deletes a rendered file, never mutates `lineage.yml`, and never retroactively adopts pre-existing `.opencode/skills/` directories. A consumer with no prior manifest is bootstrapped forward-looking (baseline from the current successful render) and sees no orphans on first run.

The central thing to understand about this release: **the seam now remembers what it rendered, so a removed overlay skill source no longer vanishes silently.** Previously, removing an overlay skill source (e.g. deleting `.vh-agent-harness/overlays/<pack>/skills/<name>/`) and re-running `vh-agent-harness update` left the previously-rendered `.opencode/skills/<name>/` live and untouched (the renderer's deliberate conservative non-deleting posture), with no record that the harness had ever produced it — invisible to both `update` and `update --dry-run`. v0.10.0 introduces a separate versioned manifest at `.vh-agent-harness/rendered-outputs.json` that records producer-aware per-FILE render provenance (producing pack + source-relative path + digest + successful render id). On each `update`, the seam loads the prior manifest, compares it against the current render, and surfaces a "Preserved orphan skill(s)" notice for any previously-rendered overlay skill whose producing SOURCE is now MISSING. Pack deselection (source still present, just not selected under `overlays:`) is explicitly NOT flagged, and project-added `.opencode/skills/` directories are never recorded so they are never flagged. The notice is **report-only**: v0.10.0 does not delete, does not add `--prune`, and does not auto-prune. (Bug reported by TrueAI against v0.9.0.)

## What changed (consumer-visible only)
| area | change | ships-via | class |
|------|--------|-----------|-------|
| preserved-orphan detection + report (fix) | The seam-apply path now distinguishes a DEFINITE orphan — a previously-rendered overlay skill whose producing source is MISSING (the pack file `skills/<name>/...` no longer exists in the producer pack, regardless of whether the pack is currently selected) — from a deselected pack (source still present, just not rendered) and from a project-added directory (never recorded, never flagged). When an orphan's destination is still present on disk, `update` and `update --dry-run` both print a "Preserved orphan skill(s)" notice naming the skill directory, its destination state (`unchanged` if the on-disk digest matches the recorded digest, `modified` otherwise), the producing pack, and the source-relative path. The notice is report-only: the file is left untouched. | Go binary + `update` (seam-apply orphan-detection path) | non-breaking (additive report surface; no deletion) |
| generated render manifest `.vh-agent-harness/rendered-outputs.json` (feat) | A new versioned JSON manifest is written under the project's `.vh-agent-harness/` directory after each successful non-dry-run `update`. It records `manifest_version` ("1"), a `successful_render_id` (the lineage last-successful update id), and per-FILE entries for overlay-rendered skills: normalized `destination_path`, `producer_kind` ("overlay_skill"), `overlay_pack`, `source_relative_path`, and `rendered_digest` (`sha256:<hex>`). It is written atomically (temp file + rename) ONLY after a successful non-dry-run apply; `--dry-run` reads and compares against it but never writes it. If no prior manifest exists, a baseline is established from the current successful render (forward-looking bootstrap — pre-existing `.opencode/skills/` directories are NOT retroactively adopted, because there is no historical proof the harness produced them). A record whose source is missing is RETAINED while its destination remains present, so the orphan keeps reporting across runs; the record is retired only when the destination is gone or the source returns. | Go binary + `update` (new `internal/renderstate/` package, written by seam-apply) | non-breaking (new additive state file; ignored by older readers) |

## How to migrate (automated)
```bash
vh-agent-harness self-update            # pull the new binary (v0.10.0)
vh-agent-harness version                # expect: 0.10.0 (<label>)
vh-agent-harness update --dry-run       # ownership-safe preview; surfaces any preserved orphan skill(s) read-only
vh-agent-harness update                 # applies platform_managed render; writes the render manifest; prints preserved-orphan notices
vh-agent-harness doctor                 # lint the result
```
No profile, ownership-class, capability, command-surface, or runtime change is required for the default consumer. Source-builders rebuilding from this tag are unaffected (no Go toolchain bump in this arc).

## What `update` handles for you
- Writes the new `.vh-agent-harness/rendered-outputs.json` manifest after a successful non-dry-run apply, recording per-file provenance for overlay-rendered skills. On the first run after upgrading (no prior manifest) it establishes a forward-looking baseline from the current render — it does NOT scan and adopt pre-existing `.opencode/skills/` directories.
- Loads the prior manifest on every `update` and `update --dry-run`, compares it against the current render, and prints a "Preserved orphan skill(s)" notice for each previously-rendered overlay skill whose producing source is now missing and whose destination is still present.
- Leaves all rendered files untouched. v0.10.0 is report-only: there is no deletion, no `--prune` flag, and no `prune` command. The renderer's conservative non-deleting posture is unchanged.
- Does not touch `lineage.yml` — the manifest is a SEPARATE file. The S1-only lineage invariant is unchanged.
- Everything tagged `local_adopted` or inactive `overlay_extension` is preserved as before.

## Watch-outs
1. **Report-only — no deletion in v1.** A preserved-orphan notice tells you a rendered overlay skill whose source was removed is still on disk; it does NOT remove it. Removing the directory is a manual operator step (or a future prune command). The v1 scope is overlay-rendered SKILLS only; commands, agents, arbitrary overlay units, and core removals are out of scope.
2. **Pack deselection is NOT an orphan.** A skill whose source still exists in its pack but whose pack is no longer listed under `overlays:` is NOT flagged — the producing source is present, just not selected. Deselection and source-removal are distinct conditions; only source-removal is reported. To stop seeing a notice, either restore the source or manually delete the rendered directory (after which the next `update` retires the record).
3. **Project-added `.opencode/skills/` directories are never flagged.** The manifest only records what the harness rendered. A skill directory a human or project created directly under `.opencode/skills/` has no manifest record and is never reported as an orphan.
4. **Forward-looking bootstrap — no retroactive adoption.** On the first run after upgrading (no prior manifest), no orphans are reported even if stale rendered overlay skills already exist on disk, because there is no historical proof the harness produced them. Orphan reporting begins only for skills rendered AFTER the manifest is established.
5. **Supersedes the v0.9.0 rollback note (line 59 (d)).** v0.9.0's rollback text claimed that re-rendering with the older binary "removes the `debugging-loop` and `tdd-loop` skills from `.opencode/skills/`". That is misleading: the renderer never deletes a rendered file when its source disappears — it preserves it (exactly the condition this release now makes visible). A binary downgrade re-renders `platform_managed` files authoritatively but does NOT remove skills whose source is absent from the older binary; it leaves them preserved on disk. `v0.9.0.md` is left unchanged as a frozen point-in-time record — this note supersedes that rollback line.
6. **The manifest is binary-written state, not a rendered corpus file.** `.vh-agent-harness/rendered-outputs.json` is written by the seam-apply path at update time, not by the corpus renderer, and carries no ownership class in `core_manifest.go`. It is safe to delete (the next `update` re-establishes a forward-looking baseline); it is safe to commit (it is deterministic for a given render).

## Rollback
A minor release is reversible by binary downgrade: reinstall the `v0.9.0` binary and `vh-agent-harness update` re-renders the `v0.9.0` corpus (the render is authoritative for `platform_managed` files). **Caveats:** rolling back (a) removes the preserved-orphan detection and report surface — `update` and `update --dry-run` no longer print orphan notices; (b) stops writing `.vh-agent-harness/rendered-outputs.json` — the file already on disk is left in place and harmlessly ignored by the v0.9.0 binary (it is not read by older code); (c) does NOT remove any rendered files — consistent with this note's supersession of the v0.9.0 rollback line 59 (d): a re-render never deletes a skill whose source is absent, it preserves it. No data migration ran, so there is nothing to undo beyond the binary reinstall and re-render. A `local_adopted` customization survives the rollback as before.

## Non-consumer changes
The arc adds a new internal package `internal/renderstate/` (the versioned manifest, per-file record normalization, validation, atomic read/write, and prior-vs-current comparison), extends `internal/substrate/apply.go`'s `ApplyReport` with a structured orphan-findings field, and wires `internal/cli/seam.go` (producer-aware record capture in the overlay render loop, prior-manifest load, orphan comparison, and atomic manifest persist after a successful non-dry-run apply). The report surfaces live in `internal/cli/guide.go` (dry-run section) and `internal/cli/update.go` (normal-update notice). These are internal to the binary's seam-apply path and carry no capability, command-surface, ownership-class, or `templates/core` change — `templates/core/` remains domain-free. The canonical consumer source is the Go binary's seam-apply path.
