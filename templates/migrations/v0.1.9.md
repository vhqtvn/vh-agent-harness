# Migration: v0.1.9

## Summary
- **Release class:** patch (semver compatible). No breaking changes.
- **Upgrade path:** binary self-update, then re-render the corpus. Fully automatic — no manual file edits required.
- **Risk:** low. Two consumer-visible changes, both non-breaking: (a) the textual `update`/`update --dry-run` summary is reworded — no domain/runtime contract change, no ownership-class change, and the summary change alone writes no file differently (the same managed files are overwritten as before; byte-identical ones are now merely skipped); and (b) a new managed core skill (`harness-operator`) is rendered into `.opencode/skills/harness-operator/` on `update`.

## What changed (consumer-visible only)
| area | change | ships-via | class |
|------|--------|-----------|-------|
| update summary | `update` and `install` now split the `outcomes:` line: managed files that are byte-identical to the freshly rendered corpus are reported as `managed-unchanged` (no write) instead of being folded into `managed-overwrite` | `update` (binary/help surface) | non-breaking |
| dry-run summary | `update --dry-run` adds an `Already current — N managed file(s) byte-identical to the corpus (no write).` line so a clean refresh is obvious | `update` (binary/help surface) | non-breaking |
| new core skill: harness-operator | a new managed skill is added to the core corpus and rendered into `.opencode/skills/harness-operator/SKILL.md` on `update`; it surfaces harness-operation guidance (the `guide → update → doctor` loop, golden rules, upgrade sequence, migration-note reading) via the native `skill` tool. Auto-discovered on next session start; no manifest/config change. Existing skills and behavior unchanged | `update` (core template render) | non-breaking / docs-only |

## How to migrate (automated)
```bash
vh-agent-harness self-update            # pull the new binary
vh-agent-harness version                # expect: 0.1.9 (<label>)
vh-agent-harness update --dry-run       # ownership-safe preview; watch for the "Already current" line
vh-agent-harness update                 # applies platform_managed + active overlay_extension
vh-agent-harness doctor                 # lint the result
```

## What `update` handles for you
- Nothing manual. The summary wording is computed at run time from the rendered-vs-live byte comparison; a plain `update` after self-update reports the new tokens. The summary change alone stages/owns/writes no file differently than before — the same managed files are overwritten as before, and byte-identical ones are now merely skipped rather than rewritten. Separately, `update` also renders the new `harness-operator` skill into `.opencode/skills/harness-operator/SKILL.md` (additive; no existing skill is touched).

## Watch-outs
1. **String-matched automation** — if a consumer's script or test asserted the old outcomes line by expecting `managed-overwrite` to account for ALL managed files, it now needs to tolerate the new `managed-unchanged` token (the two counts sum to what was previously reported as a single `managed-overwrite` figure). The label spelling is stable: `managed-unchanged`.
2. **No behavior change behind the labels** — the live tree is still churn-free: `managed-unchanged` means the seam skipped the write because the live bytes already matched the corpus, exactly the same file content `managed-overwrite` would have written. There is no semantic regression hiding in the relabel.
3. **Skill enumeration** — consumers who enumerate `.opencode/skills/` programmatically will see one new entry (`harness-operator/`) after `update`. No behavioral change; the skill is additive and only surfaces via the native `skill` tool on demand.

## Verification commands
```bash
vh-agent-harness doctor                 # expect: no new findings
vh-agent-harness update --dry-run       # expect: "Already current — ..." for an already-current install
git status                              # expect: only intended .opencode/ renders changed
```

## Rollback
Patch release is reversible: reinstall the v0.1.8 binary and `vh-agent-harness update` re-renders the prior corpus. No data migration ran, so there is nothing to undo beyond the binary reinstall. The summary-wording change touches no files. However, the new `harness-operator` skill rendered by v0.1.9 is **not** removed automatically by rolling back — the v0.1.8 corpus does not know about it, so `update` leaves it in place as a (harmless) unmanaged file. To get the exact pre-v0.1.9 skill set back, delete it manually:
```bash
rm -rf .opencode/skills/harness-operator/   # optional: remove the lingering v0.1.9 skill
git checkout HEAD -- .opencode/             # only if an unrelated render diff is unwanted
```

## Non-consumer changes
Substrate change shipped by the prior slice and referenced here for completeness: `planOutcome` in `internal/substrate/apply.go` now byte-compares the staged corpus copy against the live instance via `managedUpToDate`, routing up-to-date managed files to the new `ActionManagedNoop` (`"managed-unchanged"`) outcome instead of `ActionManagedOverwrite`, so the summary distinguishes real churn from a no-op refresh. No consumer action required.
