# Migration: v0.2.2

## Summary
- **Release class:** patch (semver compatible). No breaking changes; no profile/ownership-class change.
- **Upgrade path:** binary self-update, then re-render the corpus. Fully automatic — no manual file edits required for the default case.
- **Risk:** low. One consumer-visible change, non-breaking:
  - A new **N/A sentinel** is recognized in `project.config.json` **string-form** token fields. A token value of `none` / `n/a` / `null` / `na` (case-insensitive) now renders as empty **and counts as resolved**, so a project can opt out of a token without leaving the `{{TOKEN}}` literal in rendered output. List/array-form N/A is NOT supported (see Watch-outs).

## What changed (consumer-visible only)
| area | change | ships-via | class |
|------|--------|-----------|-------|
| N/A sentinel for token fields | A **string-form** token field in `project.config.json` whose value is one of `none`, `n/a`, `null`, `na` (matched case-insensitively, with surrounding whitespace trimmed) is now treated as a deliberate opt-out: the token resolves to the empty string and is counted as resolved by the renderer. This lets a project turn a token "off" without injecting the literal `{{TOKEN}}` placeholder into rendered files, and without leaving a stale or invented value behind. Only the STRING form is honored — the same values inside a JSON array (list form) are NOT recognized and continue to render/validate as ordinary list entries. | `update` (binary render path; binary/help surface) | non-breaking (additive opt-out) |

## How to migrate (automated)
```bash
vh-agent-harness self-update            # pull the new binary (v0.2.2)
vh-agent-harness version                # expect: 0.2.2 (<label>)
vh-agent-harness update --dry-run       # ownership-safe preview; confirm the opted-out tokens render empty
vh-agent-harness update                 # applies platform_managed + active overlay_extension; re-renders with sentinel resolution
vh-agent-harness doctor                 # lint the result
```
No profile, ownership, or capability change is required. The sentinel is interpreted by the binary renderer at render time, so adopting it is purely a matter of self-updating and re-rendering. To opt a token out, set its **string** value in `project.config.json` to one of the recognized sentinels (e.g. `"SOME_TOKEN": "none"`), then re-run `update`.

## What `update` handles for you
- After `self-update`, the v0.2.2 binary interprets `none` / `n/a` / `null` / `na` (case-insensitive, surrounding whitespace trimmed) on string-form token fields at render time: the token resolves to the empty string and is reported as resolved, so neither the `{{TOKEN}}` literal nor the sentinel text reaches rendered output. No flag, profile entry, or manifest edit is needed — the renderer reads the live `project.config.json` and applies the rule on every render (install, update, doctor's managed-drift re-render).
- A token field left at a real value is unaffected; a token field carrying the literal `{{TOKEN}}` placeholder is unaffected. The sentinel is strictly additive: it only changes the outcome for the four recognized strings.
- List/array-form token values are NOT scanned for the sentinel. A list entry such as `"none"` continues to be treated as an ordinary list member (see Watch-outs).
- Schema reconciliation on seeded files is unchanged; everything tagged `local_adopted` / inactive `overlay_extension` is preserved as before.

## Watch-outs
1. **String-form ONLY.** The sentinel is honored on a token field whose JSON value is a single string (`"TOKEN": "none"`). Placing `none` / `n/a` / `null` / `na` inside a JSON array (`"TOKEN": ["none"]`) does NOT opt the token out — the array is treated as an ordinary list value and the entry is not treated as a sentinel. If you need to opt out, use the string form.
2. **Case-insensitive, surrounding whitespace trimmed.** The four sentinels (`none`, `n/a`, `null`, `na`) match in any case, and surrounding whitespace is trimmed first — so `" none "`, `"N/A"`, and `"\tna"` all resolve as the opt-out. Suffixed or prefixed variants (e.g. `"none-x"`, `"xnone"`) are NOT sentinels and render as their literal text. Only the bare four strings (after trim) match.
3. **Empty ≠ missing.** The sentinel resolves to the EMPTY string and counts as resolved. If a downstream consumer previously asserted a non-empty value for that token, it will now see an empty value where the project opted out. This is the intended opt-out behavior; if a consumer needed a non-empty value, leave a real value in the field rather than a sentinel.
4. **List-form N/A was considered and reverted before this release.** An earlier slice in the development arc added list/array-form sentinel handling and was reverted; list-form is intentionally NOT supported in v0.2.2. Do not write a list-form sentinel expecting the same effect as the string form.

## Rollback
A patch release is reversible by binary downgrade: reinstall the v0.2.1 binary and `vh-agent-harness update` re-renders the v0.2.1 corpus (the render is authoritative for `platform_managed` files, so the sentinel resolution is reverted — a string field set to `none`/`n/a`/`null`/`na` reverts to being treated as an ordinary literal value by the v0.2.1 renderer). **Caveat:** if you set a token field to one of the sentinel strings in `project.config.json`, the v0.2.1 renderer will emit that string literally into rendered output (it does not recognize the sentinel). When rolling back, also revert any sentinel value you adopted back to its prior real value (or `{{TOKEN}}` placeholder) to avoid a literal `none`/`null` reaching rendered files. No data migration ran, so there is nothing to undo beyond the binary reinstall, re-render, and any config-value revert. A `local_adopted` customization survives the rollback (it is preserved in both directions):
```bash
git checkout HEAD -- .opencode/         # only if an unrelated render diff is unwanted
```

An earlier slice in the same arc added list/array-form sentinel handling (`49b3755`) and was reverted (`e829187`); the net shipped state is string-form only, as documented above.
