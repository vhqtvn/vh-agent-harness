# Migration: v0.4.1

## Summary
- **Release class:** patch (semver compatible). No breaking changes; no profile/ownership-class change.
- **Upgrade path:** binary self-update, then re-render the corpus. Fully automatic — no manual file edits required for the default case.
- **Risk:** low. One consumer-visible change, non-breaking:
  - The **DEFER/p2 promotion-aid** `check-defer-triggers.js` (the predicate checker referenced canonically in `AGENTS.md` as `node .opencode/scripts/check-defer-triggers.js`) was **dead-on-arrival in v0.4.0**: a wrong relative import (`../../plugins/shell-guard-core.js` — one `../` too many) made node resolve it to `<repo-root>/plugins/shell-guard-core.js` (which does not exist) and throw `ERR_MODULE_NOT_FOUND` before `main()` ran. The script never executed for any consumer who re-rendered under v0.4.0. v0.4.1 removes the broken import and inlines a zero-dep `repoRoot()` plus an ESM `__dirname` shim (mirroring `state-lib.js`), so the predicate checker loads and runs.

## How to migrate (automated)
```bash
vh-agent-harness self-update            # pull the new binary (v0.4.1)
vh-agent-harness version                # expect: 0.4.1 (<label>)
vh-agent-harness update --dry-run       # ownership-safe preview; confirm check-defer-triggers.js re-renders with the inlined repoRoot
vh-agent-harness update                 # applies platform_managed + active overlay_extension; re-renders the fixed script
vh-agent-harness doctor                 # lint the result
```
No profile, ownership, capability, command-surface, or runtime change is required. The fix is purely a re-render of one `platform_managed` script. Consumers who carry a `local_adopted` customization of `.opencode/scripts/check-defer-triggers.js` should review the migration courtesy-check diff and re-merge their customization onto the canonical fixed form so the import repair lands (see Watch-outs).

## What `update` handles for you
- Re-renders `.opencode/scripts/check-defer-triggers.js` from the v0.4.1 template: removes the broken `import { repoRoot } from "../../plugins/shell-guard-core.js";` and inlines the self-contained `repoRoot()` plus the `fileURLToPath`-based `__dirname`/`__filename` shim. After `update`, `node .opencode/scripts/check-defer-triggers.js` loads and runs instead of throwing `ERR_MODULE_NOT_FOUND`.
- No profile, ownership-class, capability, agent, skill, command-surface, or runtime change. No `opencode.jsonc` permission change. No schema-reconciliation change.
- Everything tagged `local_adopted` or inactive `overlay_extension` is preserved as before; a customized `check-defer-triggers.js` triggers the standard courtesy check rather than being silently clobbered.

## Watch-outs
1. **The bug shipped in v0.4.0.** Every consumer who re-rendered the corpus under v0.4.0 received the broken script. Re-rendering under v0.4.1 repairs it. Because `check-defer-triggers.js` is **promoter-use-only and never wired into any blocking path** (no commit hook, no gate, no automated invocation), the DOA bug had no operational effect beyond the promotion-aid being unavailable — no safety, correctness, or release-blocking regression occurred. Nothing needs to be undone beyond re-rendering.
2. **Customized `check-defer-triggers.js`.** If a previous `local_adopted` edit customized the script, the migration courtesy check fires and `update` will NOT silently clobber it. Review the diff and re-merge your customization onto the canonical fixed form so the import repair lands.
3. **`node` is required to run the script.** `check-defer-triggers.js` is plain Node ESM and is invoked manually by promoters (`node .opencode/scripts/check-defer-triggers.js`). The new regression guard skips when `node` is not on PATH; the script itself is only a concern for promoters running the predicate check.

## Rollback
A patch release is reversible by binary downgrade: reinstall the v0.4.0 binary and `vh-agent-harness update` re-renders the v0.4.0 corpus (the render is authoritative for `platform_managed` files). **Caveat:** rolling back re-renders the broken script, so `node .opencode/scripts/check-defer-triggers.js` throws `ERR_MODULE_NOT_FOUND` again (the predicate checker returns to dead-on-arrival). Because the script is promoter-use-only and never blocking, the only effect of rolling back is losing the promotion-aid — no safety or correctness regression. No data migration ran, so there is nothing to undo beyond the binary reinstall and re-render. A `local_adopted` customization survives the rollback (it is preserved in both directions):
```bash
git checkout HEAD -- .opencode/         # only if an unrelated render diff is unwanted
```

## Non-consumer changes
The fix landed in commit `201123d` `fix(scripts): make check-defer-triggers.js load under ESM`, applied identically to both the template source (`templates/core/.opencode/scripts/check-defer-triggers.js`) and the rendered copy (`.opencode/scripts/check-defer-triggers.js`) via `make update`. A regression guard `internal/cli/check_defer_triggers_test.go` (`TestCheckDeferTriggersScriptLoads`) was added in the same commit: a hermetic node smoke test that copies the template script into a `t.TempDir()` at `<scratch>/.opencode/scripts/`, runs `node <dst> --tasks <missing>` with `cmd.Dir=scratch` and a 30s timeout, and asserts the combined output contains no `ERR_MODULE_NOT_FOUND` / `Cannot find module`, that the exit code is 0, and that stdout contains the `no tasks dir` and `never blocking` degrade announcements. The test skips when `node` is not on PATH and uses a local `findModuleRoot` helper that walks to `go.mod`. The test is internal-only and needs no consumer action beyond this note.
