# Migration: v0.4.2

## Summary
- **Release class:** patch (semver compatible). No breaking changes; no profile/ownership-class change; no `opencode.jsonc` permission change.
- **Upgrade path:** binary self-update, then re-render the corpus. Fully automatic — no manual file edits required.
- **Risk:** low. One consumer-visible change, non-breaking (a security tightening):
  - **F1 — closed the `vh-agent-harness exec git <global-flag> <mutation>` commit-gate bypass.** A wrapped git mutation routed past a leading global flag — `vh-agent-harness exec git --no-pager commit`, `exec git -C /x push`, `exec git --git-dir=/x commit` — previously slipped past the commit-gate: the `git-mutation-bypass` adjacency regex (`\bgit\s+<mutation-verb>`) could not match a flag between `git` and the verb, and the harness branch returned `{action:"allow"}` after only the `isGateWrapperInDevShExec` check. The git-mutation parser (`walkGitGlobals`) never ran for wrapped payloads. v0.4.2 denies these forms at TWO layers (defense in depth): the A2 JS source-of-truth parser (`detectWrappedGitMutation` in `shell-guard-core.js`, reusing `walkGitGlobals` + `GIT_MUTATION_VERBS`) and the A1 Go binary backstop (`denyExecGitMutationPayload` in `internal/cli`, reusing `execro.GitVerbPastGlobals` + `permconfig.GitMutationVerbsSet`). Credit: TrueAI (responsible disclosure).

## What changed (consumer-visible only)
| area | change | ships-via | class |
|------|--------|-----------|-------|
| `vh-agent-harness exec git <global-flag> <mutation>` now denied | A wrapped git mutation routed past a leading global flag is now denied at both the JS shell-guard gate (A2) and the Go binary (A1, before the JS bridge even runs). The mutation must route through the commit-gate wrapper / committer agent as before. Now-denied forms include `vh-agent-harness exec git --no-pager commit`, `exec git -C /x push`, `exec git --git-dir=/x commit`, `exec git --no-pager push`, and the adjacent regression `exec git commit`. Unaffected: read-only git through exec (`exec git --no-pager status`, `exec git -C /repo log`) and ALL non-git exec mutations (`exec mkdir`, `exec pytest`, `exec npm test`, `exec bash -c '...'`) — A1/A2 are git-mutation-scoped only and do NOT default-deny (they intentionally do not call `execro.Classify`, which is curated-allowlist/default-deny and would break the legitimate mutating exec surface). | `update` (core template: `templates/core/.opencode/plugins/shell-guard-core.js` → rendered `.opencode/plugins/shell-guard-core.js`) + Go binary | non-breaking (security tightening — closes a bypass; no previously-allowed legitimate command is denied) |

## How to migrate (automated)
```bash
vh-agent-harness self-update            # pull the new binary (v0.4.2)
vh-agent-harness version                # expect: 0.4.2 (<label>)
vh-agent-harness update --dry-run       # ownership-safe preview; confirm shell-guard-core.js re-renders with detectWrappedGitMutation
vh-agent-harness update                 # applies platform_managed + active overlay_extension; re-renders the fixed plugin
vh-agent-harness doctor                 # lint the result
```
No profile, ownership, capability, command-surface, or runtime change is required. The fix is a re-render of one `platform_managed` plugin plus the new binary's Go backstop. Consumers who carry a `local_adopted` customization of `.opencode/plugins/shell-guard-core.js` should review the migration courtesy-check diff and re-merge their customization onto the canonical fixed form so the A2 parser lands (see Watch-outs).

## What `update` handles for you
- Re-renders `.opencode/plugins/shell-guard-core.js` from the v0.4.2 template: adds `detectWrappedGitMutation(normalizedCmd, cwd)` and wires it into `evaluate()`'s harness branch before the allow. It tokenizes the wrapped payload, skips harness wrapper flags (`--service`/`--workdir`/`-w`/`--tty`/`-t`/`--` for `exec`; optional `--` for `exec-ro`), and if the payload starts with `git` calls the EXISTING `walkGitGlobals` to extract the verb past global flags; if the verb is in `GIT_MUTATION_VERBS` it returns a deny mirroring the `git-mutation-bypass` rationale. Reuses `walkGitGlobals` + `GIT_MUTATION_VERBS` — no new parser, no regex, no local verb copy.
- The Go binary backstop (`denyExecGitMutationPayload`) ships in the binary itself (not the rendered tree). It runs in `runExec` after the `pre_exec` hook and before `evaluateGate`, denying a bare git-mutation payload before the JS bridge is invoked.
- No profile, ownership-class, capability, agent, skill, command-surface, or runtime change. No `opencode.jsonc` permission change. No schema-reconciliation change.
- Everything tagged `local_adopted` or inactive `overlay_extension` is preserved as before; a customized `shell-guard-core.js` triggers the standard courtesy check rather than being silently clobbered.

## Watch-outs
1. **The bypass was real and load-bearing.** Before v0.4.2, an agent (or operator) invoking `vh-agent-harness exec git --no-pager commit` (or any global-flag-prefixed mutation) bypassed the committer-exclusive commit-gate entirely — the mutation reached the backend with no gate. v0.4.2 closes it at two independent layers; either layer alone is sufficient. If your workflows relied on this form (they should not have — `AGENTS.md` routes all git writes through the committer agent / `commit-gate.sh`), switch to the blessed committer flow.
2. **Customized `shell-guard-core.js`.** If a previous `local_adopted` edit customized the plugin, the migration courtesy check fires and `update` will NOT silently clobber it. Review the diff and re-merge your customization onto the canonical fixed form so `detectWrappedGitMutation` lands.
3. **Nested shell strings are OUT OF SCOPE.** `vh-agent-harness exec bash -c 'git commit …'` is NOT parsed by A1/A2's wrapped-mutation guard (neither parses nested shell strings). It remains governed by the EXISTING forbidden-pattern chain-guard scan (`bash -c` / `sh -c` / `eval` + git-mutation inspection), which already denies it. v0.4.2 changes nothing about that path.
4. **Read-only git and non-git exec are unaffected.** `exec git --no-pager status`, `exec git -C /repo log`, `exec mkdir`, `exec pytest`, `exec npm test`, `exec bash -c 'echo hi'` all continue to pass through to the backend exactly as before. A1/A2 are scoped to git MUTATIONS only — they do not default-deny and do not call `execro.Classify`.

## Rollback
A patch release is reversible by binary downgrade: reinstall the v0.4.1 binary and `vh-agent-harness update` re-renders the v0.4.1 corpus (the render is authoritative for `platform_managed` files). **Caveat:** rolling back re-introduces the F1 bypass — `vh-agent-harness exec git --no-pager commit` would again reach the backend ungated. Do not roll back if the bypass is a live concern. No data migration ran, so there is nothing to undo beyond the binary reinstall and re-render. A `local_adopted` customization survives the rollback (it is preserved in both directions):
```bash
git checkout HEAD -- .opencode/         # only if an unrelated render diff is unwanted
```

## Non-consumer changes
The fix landed as a security-hardening slice closing the F1 bypass via the decided A1+A2 design (command parsing, not regex). A2 (JS source-of-truth): `templates/core/.opencode/plugins/shell-guard-core.js` gains `detectWrappedGitMutation` (reuses `walkGitGlobals` + `GIT_MUTATION_VERBS`, no regex, no local verb copy) wired into `evaluate()`'s harness branch before the allow.
