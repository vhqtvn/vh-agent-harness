# Migration: v0.2.1

## Summary
- **Release class:** patch (semver compatible). **SECURITY release.** No breaking changes; no profile/ownership-class change.
- **Upgrade path:** binary self-update, then re-render the corpus. Fully automatic — no manual file edits required for the default case.
- **Risk:** low–medium. Three consumer-visible changes, all non-breaking:
  - A shell-guard bypass is CLOSED: `git --no-pager <mutation>` (and `--paging=no`, any safe global flag prefixing a mutation verb) no longer slips past `git-mutation-bypass`. Read-only single-flag prefixes (`git --no-pager log`) are now prompt-free.
  - `.opencode/scripts/commit-gate.sh status` moves from the `gate` permission group to `readonly`, so the pure-read lock probe is prompt-free for ALL agents (including gate-exempt ones).
  - The committer message-staging contract changes: the heredoc `stage-message` form is banned (its multi-line body tripped `git-mutation-bypass` and the carve-out chain-guard). The committer now authors the message via the Write tool and passes it with `acquire --message-file`.

## What changed (consumer-visible only)
| area | change | ships-via | class |
|------|--------|-----------|-------|
| shell-guard security closer | `git --no-pager commit` (and `--paging=no`, any safe global flag prefixing a mutation verb) previously slipped past the `git-mutation-bypass` forbidden scan because the regex requires verb-adjacency (`\bgit\s+(add\|commit\|...)\b`). `normalizeGitGlobalFlags()` now strips the safe global flags (`--no-pager`, `--paging=no`) on the reconstruction and RE-RUNS the forbidden-pattern deny scan, so `git --no-pager commit` is correctly DENIED. Mutations are caught regardless of how many safe flags prefix the verb. | `update` (core template: `shell-guard-core.js`) | non-breaking (security hardening) |
| `git --no-pager` readonly UX | 12 new config entries (`git --no-pager {diff,log,show,grep,blame,ls-tree,status,ls-files,check-ignore,cat-file,show-ref,rev-parse} *`) kill the operator prompt for the common single-flag read-only case. These are config-table defense-in-depth; the mutation safety closer above lives in shell-guard, not here. | `update` (permconfig emit → `opencode.jsonc` bash block) | non-breaking |
| `commit-gate.sh status` group | `status` is a pure-read lock probe (reads lock/session metadata only, emits JSON). It moves from the `gate` group to `readonly`, so it is prompt-free for ALL agents including the gate-exempt ones (build, coordination, project-coordinator, docs-steward). The mutation verbs (`acquire`, `commit`, `release`, `heartbeat`, `revert`, `stage-message`) remain in the `gate` group — committer-only, unchanged. No gate-exempt invariant change. | `update` (permconfig emit → `opencode.jsonc` bash block) | non-breaking |
| committer message-staging contract | The heredoc `stage-message` form is BANNED: a multi-line command body contains a newline, and (a) `git-mutation-bypass` scans the raw command string including the heredoc body, so a commit message mentioning any git verb (e.g. "fix: git rebase workflow") matched the forbidden regex; (b) the `ALLOW_IF_GIT_MUTATION` chain-guard carve-out refuses any string containing a statement separator (`;`, newline, CR, `&`, `\|`, backtick, `$(`, process-sub), so the carve-out never applied. Message = file = DATA; command = single-line CONTROL. | `update` (core template: `committer.md`, gated-commit `SKILL.md`, `commit-gate.sh`) | non-breaking |
| committer scoped edit | The committer gains the ONLY object-form edit in the corpus: `edit: {"*":"deny","tmp/commit-gate-message/**":"allow"}` (deny-first / allow-last). This widens edit to exactly the one path the message-file flow writes; every other path denies. All other agents' edit decisions are unchanged. | `update` (permconfig emit → `opencode.jsonc`) | non-breaking |

## How to migrate (automated)
```bash
vh-agent-harness self-update            # pull the new binary (v0.2.1)
vh-agent-harness version                # expect: 0.2.1 (<label>)
vh-agent-harness update --dry-run       # ownership-safe preview; watch the shell-guard + permconfig renders
vh-agent-harness update                 # applies platform_managed + active overlay_extension; ships the security closer + new contract
vh-agent-harness doctor                 # lint the result
```
No profile, ownership, or capability change is required — the security closer, the readonly-group moves, and the committer message-staging contract are all delivered by re-rendering the corpus. Consumers with a `local_adopted` customization of `shell-guard-core.js`, `committer.md`, or the gated-commit `SKILL.md` should review the courtesy-check diff and re-merge onto the canonical form (see Watch-outs).

## What `update` handles for you
- Re-renders `.opencode/plugins/shell-guard-core.js` (and the node shim that imports it) with `normalizeGitGlobalFlags()`, closing the `git --no-pager <mutation>` bypass.
- Emits the 12 `git --no-pager <readonly-sub> *` entries and the `.opencode/scripts/commit-gate.sh status` entry into the `readonly` group of the top-level `permission.bash` block (permconfig tables are the single canonical source).
- Re-renders `.opencode/scripts/commit-gate.sh`, `.opencode/agents/committer.md`, and `.opencode/skills/gated-commit/SKILL.md` with the `--message-file` message-staging contract (the `acquire`/`commit`/`stage-message` subcommands already accept `--message-file FILE`).
- Emits the committer's scoped object-form edit (`{"*":"deny","tmp/commit-gate-message/**":"allow"}`); every other agent's edit decision is byte-identical to before.
- Reconciles schema on seeded files; preserves everything tagged `local_adapted` / inactive `overlay_extension` unchanged.

## Watch-outs
1. **Two-flag read-only prefixes are safely over-blocked (hard-denied).** `git --no-pager --paging=no log` is NOT covered by the 12 prompt-free config entries (the matcher compares tokens positionally and only single-flag forms are listed). Because `--no-pager` is itself a member of `GIT_READONLY_SUBCOMMANDS` (a side-effect of the 12 `git --no-pager <sub> *` config entries contributing `parts[1]`), the routing-hint ask-branch guard `!has(blocked[1])` is false, so this two-flag form falls through to the final **hard-deny** — it is blocked, not prompted, not allowed. Use the single-flag form (`git --no-pager log`). This is a read-only UX limitation only — mutations are caught regardless of flag count, because the re-scan normalizer strips the whole safe-flag run before re-running `git-mutation-bypass`.
2. **Global-flag scope is intentionally narrow.** Only `--no-pager` and its exact synonym `--paging=no` are normalized. Other git global flags (`--git-dir`, `--work-tree`) take path arguments that could escape the repo and are deliberately left between `git` and the subcommand, so they do NOT get a prompt-free entry and will prompt. Conservative by design.
3. **Customized shell-guard / committer files.** If a previous `local_adopted` edit customized `.opencode/plugins/shell-guard-core.js`, `.opencode/agents/committer.md`, or `.opencode/skills/gated-commit/SKILL.md`, the migration courtesy check fires and `update` will NOT silently clobber it. Review the diff and re-merge your customization onto the canonical form so the security closer and the new message-staging contract both land.
4. **Security release — prefer not to roll back.** See Rollback: a binary downgrade re-opens the Q1 bypass. Roll back only with a compensating control.

## Rollback
A patch release is reversible by binary downgrade: reinstall the v0.2.0 binary and `vh-agent-harness update` re-renders the v0.2.0 corpus (the render is authoritative for `platform_managed` files, so the security closer, the 12 readonly entries, the `status` group move, and the committer message-staging contract are all reverted to their v0.2.0 form). **Caveat: this re-opens the `git --no-pager <mutation>` bypass** — roll a security release back only with a compensating control, and prefer forward-fixing instead. No data migration ran, so there is nothing to undo beyond the binary reinstall and re-render. A `local_adopted` customization survives the rollback (it is preserved on both directions):
```bash
git checkout HEAD -- .opencode/         # only if an unrelated render diff is unwanted
```
