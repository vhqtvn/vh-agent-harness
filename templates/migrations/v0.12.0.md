# Migration: v0.12.0

## Summary
- **Release class:** minor (semver compatible). The `v0.11.0..HEAD` arc (13 commits) is additive on the shipped core corpus and the `doctor` diagnostic surface, with **zero `!:` subjects and no `BREAKING CHANGE:` footers** → MINOR bump; not patch/major. The arc carries four `feat:` commits clustered into three consumer-visible capabilities: the opt-in `core/media-perception` capability (`364c612` + `80d4eaf`), the `doctor` `subagent-depth` check #11 (`6abb3f6`), and the `doctor` `auto-gate-ignore` check with `.gitignore` seeding (`d4cd818`). Every consumer-visible change is additive/opt-in: the capability is gated off by default (the default consumer's `opencode.jsonc` is byte-identical unless they opt in), and the two new `doctor` checks are inert unless their trigger conditions exist (an insufficient `subagent_depth` config; a never-commit config file that is effectively unignored or tracked). Last tag = `v0.11.0`; intended version = `v0.12.0`.
- **Upgrade path:** binary self-update, then re-render the corpus. Fully automatic for the default consumer. The two new `doctor` checks and the `media-perception` deny-all task-map pin ship in the Go binary (acquired by `self-update`); the `media-perception` agent/skill/caller-edges, the `auto-gate-ignore` `.gitignore` seed, and the `README.agent.md` updates re-render through `vh-agent-harness update`. No profile, ownership-class, capability, or runtime change is required for the default consumer — `core/media-perception` is opt-in and unions onto the preset only when explicitly selected. `templates/core/` remains domain-free (token-only).
- **Risk:** low. Every consumer-visible change is additive and opt-in. `core/media-perception` is OFF by default (capability-gated in `opencode.jsonc.tmpl` via `{{- if .capabilities.media_perception }}`); selecting it adds exactly one read-only perception leaf with a deny-all task map (pinned in `tables.go` with a regression test so the emitter's wildcard-merge optimization cannot silently drop it). The `subagent-depth` check is WARN-tier (not FAIL) and SKIPs unparseable project files, so a malformed hand-edited config cannot fail `doctor`. The `auto-gate-ignore` check is inert unless a never-commit config file exists, and its secret-handling preserves the D6 invariant (a literal apiKey is never emitted into `doctor` output — only the file path and rotate/revoke guidance). The `.gitignore` seed is scoped (`*.local.json` wildcard + exact `auto-gate-llm.json`, kept separate per v0.10.0) and only seeds `templates/core/.gitignore` (managed); it never injects into a project-owned `.gitignore`.

## Watch-outs
1. **`core/media-perception` is OPT-IN and OFF by default.** The default consumer's `opencode.jsonc` does not render the `media-perception` agent, its skill, or the four caller edges unless the profile selects `capabilities: [- core/media-perception]`. This is intentional and validates the capability gate end-to-end. Opting in is additive (unions onto the preset); it never removes existing agents or weakens an existing deny rule.
2. **The `media-perception` leaf carries a deny-all task map.** It is pinned in `tables.go` (`media-perception: [{*,Deny}]`) with regression test `TestEmit_MediaPerceptionLeafWildcardAlwaysPreserved` so the emitter's wildcard-merge optimization cannot silently drop it. The leaf is read-only and cannot be delegated implementation work; callers perceive media through it, they do not commission edits.
3. **`subagent-depth` is WARN-tier, not FAIL.** The check warns when `subagent_depth` is unset or below 6 (the deepest documented chain `coordination -> researcher -> debate -> planner -> build -> committer`). The harness still functions at a lower depth — OpenCode caps delegation, it does not fail. The check SKIPs unparseable project files rather than failing `doctor` on a malformed hand-edited config.
4. **`auto-gate-ignore` is inert unless a never-commit config file exists.** The check follows v0.7.0 discipline: it does nothing unless the overlay is selected OR a relevant config file (`auto-gate-llm.json`, `*.local.json` companions) is present. Consumers not using the auto-classifier-pilot overlay will see no new `doctor` output from this check. A FAIL on a tracked file with a literal `apiKey` emits rotate/incident guidance but **never the key value** (D6 invariant).
5. **The `.gitignore` seed is scoped and managed only.** It adds `*.local.json` (scoped wildcard) and `auto-gate-llm.json` (exact) to `templates/core/.gitignore` (a `platform_managed` file). It does NOT inject rules into a project-owned `.gitignore` (mechanism C was explicitly rejected to preserve the `project_owned` contract). Existing installs that already ignore these paths are unaffected; the seed protects greenfield installs.

## Rollback
A minor release is reversible by binary downgrade: reinstall the `v0.11.0` binary and `vh-agent-harness update` re-renders the `v0.11.0` corpus (the render is authoritative for `platform_managed` files). **Caveats:** rolling back (a) removes the `subagent-depth` and `auto-gate-ignore` `doctor` checks and the `media-perception` deny-all task-map pin — all are binary-produced, so the older binary simply does not emit them; (b) re-renders the older `templates/core/.gitignore` (without the `auto-gate-llm.json` / `*.local.json` seed) — consumers who relied on the seed should re-add the rules to their project-owned `.gitignore` manually; (c) does NOT un-select `core/media-perception` if the profile was edited to opt in — the profile is `project_owned`/armed-merged and survives the rollback, but the v0.11.0 binary does not know the capability and `update --dry-run` will report it as unresolvable until the profile entry is removed or the binary is upgraded again; (d) leaves `.vh-agent-harness/rendered-outputs.json` and `lineage.yml` semantics unchanged. No data migration ran, so there is nothing to undo beyond the binary reinstall and re-render. A `local_adopted` customization survives the rollback as before.
