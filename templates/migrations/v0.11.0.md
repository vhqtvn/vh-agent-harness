# Migration: v0.11.0

## Summary
- **Release class:** minor (semver compatible). The `v0.10.0..HEAD` arc (14 commits) is additive on the consumer CLI surface and the shipped plugin corpus, with **zero `!:` subjects and no `BREAKING CHANGE:` footers** → MINOR bump; not patch/major. The arc carries three `feat:` commits — `143ffba` `feat(cli/update): --prune-orphans`, `67bc3d4` `feat(skill): skill list/validate + doctor skill-health check`, `2f24ca9` `feat(guide): progressive-disclosure onboarding summary` — plus a consumer-visible `cli` change (`dad68bd3`, cobra command grouping in `--help` + a command-surface drift gate) and a `coordination-hints` plugin feature (`73cdd89`, the C10 command-repetition signal trigger that ships through the embedded JS corpus). The Phase-5 roster flip (the canonical BREAKING case) is NOT in this arc — it shipped earlier in v0.2.0. All consumer-visible changes are additive/opt-in with defaults preserved. Last tag = `v0.10.0`; intended version = `v0.11.0`.
- **Upgrade path:** binary self-update, then re-render the corpus. Fully automatic for the default consumer. The new CLI commands/flags and the `--help` grouping ship in the Go binary (acquired by `self-update`); the README.agent.md restructure and the coordination-hints C10 plugin re-render through `vh-agent-harness update`. No profile, ownership-class, capability, or runtime change for the default consumer. `templates/core/` remains domain-free (token-only).
- **Risk:** low. Every consumer-visible change is additive and opt-in. `update --prune-orphans` is the only destructive surface, and it is OPT-IN (default `update` is byte-for-byte unchanged: orphans are still only listed, never touched), path-traversal-contained (B1 guard), refuse-safe on operator-edited orphans, and never deletes project-owned files. The new `doctor` skill-validity check only FAILs on genuinely invalid frontmatter or a half-deleted skill; valid existing skills are unaffected. The coordination-hints C10 toast fires at most once per session/key through the existing anti-spam dedup.

## Watch-outs
1. **`--prune-orphans` is OPT-IN and destructive.** The default `update` is unchanged (orphans listed, not touched). With the flag, only `DestUnchanged` (byte-identical) orphans are deleted; operator-edited (`DestModified`) orphans are REFUSED and reported, never deleted; already-gone (`DestMissing`) orphans are skipped. It composes with `--dry-run` (reports what it would delete, deletes nothing). The B1 path-traversal guard refuses absolute `DestinationPath` and any `..`-escape in BOTH live and dry-run modes. It never deletes project-owned files (`isProjectOwnedOrphan`), and a classifier error is refuse-safe. A `--prune-orphans` deletion is NOT reversible by a later rollback — the bytes are gone (see Rollback).
2. **coordination-hints C10 fires at most once per session/key.** The command-repetition toast needs >=3 repetitions of the same normalized command SHAPE in a session and is deduped once per session/key through the existing anti-spam. Normalization is intentionally START STRICT (collapses obviously-volatile path/number/quoted tokens); only clearly-volatile tokens collapse, so distinct commands are not merged. Prefer false-negatives over toast-spam; the threshold bounds firing.
3. **`--help` grouping is a presentation reorganization, not a semantic change.** Commands now render under Lifecycle / Orientation / Health & Diagnostics / Runtime. No command name, flag, or behavior changed; but scripts that textually parse `--help` output may see different ordering/grouping.
4. **`skill validate` and the `doctor` skill-validity check will FAIL on genuinely invalid skills.** The check FAILs on invalid frontmatter (bad/unknown keys, missing name/description) AND on a missing `SKILL.md` (it enumerates skill dirs independently so a half-deleted skill is caught). Valid existing skills are unaffected; if `doctor` newly reports a skill problem after upgrading, the skill was already non-canonical — fix its frontmatter or remove it.
5. **The migration-note ownership contract was clarified in this arc (harness-internal).** The `fix(release)` commit at HEAD (`1825d7e`) reconciles who authors the migration note (the releaser, at cut time) between the shipped release pack and the dogfood-local readiness gate. It is release-tooling plumbing with a regression test; it adds NO consumer command surface and ships no behavioral change to consumers. It is documented here only so the rationale for this very note's existence is on the record.

## Rollback
A minor release is reversible by binary downgrade: reinstall the `v0.10.0` binary and `vh-agent-harness update` re-renders the `v0.10.0` corpus (the render is authoritative for `platform_managed` files). **Caveats:** rolling back (a) removes the `--prune-orphans` flag, the `skill` command, the `doctor` skill-validity check, the `guide` progressive-disclosure summary, and the `--help` command grouping — all are binary-produced, so the older binary simply does not emit them; (b) re-renders the older `coordination-hints` plugin, retiring the C10 command-repetition toast (the `command.executed` branch early-returns again); (c) does NOT undo any deletion already performed by an earlier `update --prune-orphans` — deleted byte-identical orphan files are gone and are not restored by a binary downgrade (re-create them from source or re-add the producing overlay pack); (d) leaves `.vh-agent-harness/rendered-outputs.json` semantics unchanged (v0.10.0 introduced it; both versions read/write it compatibly). No data migration ran, so there is nothing to undo beyond the binary reinstall and re-render. A `local_adopted` customization survives the rollback as before.
