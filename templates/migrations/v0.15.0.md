# Migration: v0.15.0

## Summary
- **Release class:** MINOR (semver compatible). The `v0.14.0..HEAD` arc (15 commits) headline is **machine-enforced release-safety** — a doctor pre-flight gate (G0c) wired into the release-tag wrapper, a new doctor check (#13 `staged-errata-content`) that refuses to release when a staged erratum's correction has not been carried into the about-to-release note, and a `vh-agent-harness release inject-errata` subcommand that injects each staged erratum into the note AND flips its card `staged → completed` in a two-phase fail-closed pass (the note is written FIRST, and only then are the cards flipped; zero copy-paste). This arc exists precisely because a v0.12.0 media-perception erratum was missed across v0.13.0 → v0.13.1 → v0.14.0 (3 releases); the new gate makes such a miss mechanically impossible for v0.15.0+. The arc carries **nine `feat` commits**, **one `fix`**, **zero `!:` subjects**, and **no `BREAKING CHANGE:` footers** → MINOR bump; not patch/major. The headline is `da09e9e` (release-safety enforcement wiring); the secondary arc is the **`read_only` permconfig migration** (`ba68c76`, `5fcd295`, `2326fe5`, `b51e440`, `5077417`, `b652311`) introducing a first-class read-only HarnessPolicy for 9→17→18 read-only specialist agents. Last tag = `v0.14.0`; intended version = `v0.15.0`.
- **Upgrade path:** binary self-update, then re-render the corpus. Fully automatic for the default consumer. The release-safety enforcement ships in the Go binary (doctor check #13 + the `release inject-errata` machine step + the `releaser` agent's newly-documented G0c/inject-errata steps) plus the repo-local dogfood `scripts/release-tag.sh` wrapper (G0c gate — consumers author their own wrapper). The read-only policy, the migrated agent permission packs, the capability-owned CoreOutputs filtering, the §4.1 claim/verifier closure kernel, and doctor check #12 all ship in the Go binary + embedded corpus (acquired by `self-update`) and re-render through `vh-agent-harness update`. `templates/core/` remains domain-free (token-only).
- **Risk:** LOW. The release-safety gates are **additive and fail-closed** — they can only REFUSE a tag or refuse to release, never force one; the G0c doctor gate runs the existing `vh-agent-harness doctor` (all 13 checks) and refuses on non-HEALTHY. The `read_only` HarnessPolicy is a deny-first narrowing: read-only specialists (repo-explorer, researcher, planner, docs-steward, commit-message, commit-reviewer, ship-review, debate, media-perception) keep exactly their read-only tools plus the 28-entry canonical read-only allow inventory; they LOSE any previously-overbroad write/exec grants. Default consumers without the `release` overlay selected see no release-tag enforcement change. The capability-owned CoreOutputs filtering (`ae5b30d`) is a correctness fix that makes capability-owned files respect capability selection (previously they always rendered — see the v0.12.0 erratum). No public API changes; no Go toolchain bump.

## Watch-outs
1. **The release-safety gates are fail-closed and dogfood-gated.** G0c (the doctor pre-flight in `scripts/release-tag.sh`), check #13 (`staged-errata-content`), and `release inject-errata` are release-time enforcement for this repo's dogfood. Consumers selecting the `release` overlay get the `releaser` ceremony steps and the `release inject-errata` subcommand; the `scripts/release-tag.sh` wrapper itself is project-local dogfood infra that consumers author for themselves.
2. **`read_only` is a deny-first narrowing.** Read-only specialists now keep ONLY their read-only tools plus the 28-entry canonical read-only allow inventory. If you had relied on an overbroad write/exec grant to one of these agents (repo-explorer, researcher, planner, docs-steward, commit-message, commit-reviewer, ship-review, debate, media-perception), that grant is now correctly denied — re-grant it explicitly at the permission-pack layer if the write/exec was genuinely intended.
3. **Capability-owned files now respect capability selection (`ae5b30d`).** Previously the `media-perception` agent prompt + skill rendered on disk by every `update` even when the capability was not selected (only the `opencode.jsonc` registration was gated). They now render only when selected. If your tooling asserted the presence of an unselected capability's prompt/skill files on disk, that assertion will now fail — select the capability or update the assertion.
4. **`release inject-errata` is the ONLY sanctioned way to carry a staged erratum into a note.** Do not copy-paste erratum text by hand: the machine step both injects the section AND flips the card `staged → completed` in a two-phase fail-closed pass (note written first, then cards flipped), and doctor check #13 verifies the injected content matches the staged card. A hand-paste leaves the card `staged` and check #13 refuses the release.
5. **Two distinct release-evaluator failure classes (unchanged from v0.13.0).** The DEFER manifest evaluator still distinguishes "release-relevant finding requires disposition" (classification=blocker, exit 1) from "manifest missing/malformed/stale" (classification=evaluator-error, exit 2). Override cannot cure evaluator-errors — only repair of the committed manifest can.

## Rollback
A minor release is reversible by binary downgrade: reinstall the `v0.14.0` binary and `vh-agent-harness update` re-renders the `v0.14.0` corpus (the render is authoritative for `platform_managed` files). **Caveats:** rolling back (a) removes the release-safety enforcement wiring — the G0c doctor gate, doctor check #13, and the `release inject-errata` subcommand revert, so the v0.12.0 erratum-leak class is once again possible; (b) reverts doctor check #12 `defer-liveness`; (c) reverts the `read_only` HarnessPolicy — the read-only specialists regain their v0.14.0 permission packs (any overbroad grant returns); (d) reverts the capability-owned CoreOutputs filtering — capability-owned files render on disk by every `update` again regardless of selection; (e) reverts the §4.1 claim/verifier closure kernel; (f) does NOT un-write any committed manifest, readiness artifact, or migration note — seeded files survive the rollback, but the v0.14.0 binary consults them per its own flow. No data migration ran, so there is nothing to undo beyond the binary reinstall and re-render. A `local_adopted` customization survives the rollback as before.

## Errata for v0.12.0

## What was wrong

The v0.12.0 migration note (`templates/migrations/v0.12.0.md`) made two
categories of false claim about the `core/media-perception` capability.

### 1. File rendering (lines 11, 28, 32)

**Claimed:** When `core/media-perception` IS selected, the agent prompt and
skill files additionally render. When NOT selected, none of these render.

**Actual behavior:** The agent prompt (`.opencode/agents/media-perception.md`)
and skill file (`.opencode/skills/media-perception/SKILL.md`) render on disk
as managed baseline by every `update`. The core renderer walks the whole
`templates/core/` source tree unconditionally, regardless of capability
selection. Only the `media-perception` agent registration block and the four
`media-perception: allow` caller edges in `opencode.jsonc` are capability-gated
via `{{- if .capabilities.media_perception }}`. When the capability is not
selected, those `opencode.jsonc` entries do not render and the agent is not
callable, but the prompt and skill files remain on disk.

Note: as of v0.15.0+, capability-owned CoreOutputs filtering was added so that
capability-owned files DO respect capability selection. The v0.12.0 claim was
false at the time of that release and remained false through v0.14.0.

### 2. YAML capability-list syntax (lines 11, 23, 32)

**Wrong:** `capabilities: [- core/media-perception]`

**Correct:** `capabilities: [core/media-perception]`

A leading `- ` inside an inline YAML sequence produces the wrong string value.
Use either the inline form `[core/media-perception]` or a multiline block:

```yaml
capabilities:
  - core/media-perception
```

## Impact

Low. The capability gate at the `opencode.jsonc` registration surface is
correct — the agent is not callable when unselected. The files-on-disk
behavior is cosmetic (inert prompt/skill present but unregistered).
