# Migration: v0.5.0

## Summary
- **Release class:** minor (semver compatible). No breaking changes; no profile/ownership-class change. The v0.4.1..HEAD arc carries two `feat:` commits (`exec-sandbox`, permconfig `tmp/**` allow) → MINOR bump; no breaking change → not v1.0.0. Last tag = v0.4.1; intended version = v0.5.0.
- **Upgrade path:** binary self-update, then re-render the corpus. Fully automatic — no manual file edits required for the default case.
- **Risk:** low. Four consumer-relevant changes in the arc, all non-breaking (one is a security tightening, one is an additive narrow permission carve-out, one is an additive new command, one is a source-build toolchain bump):
  - **F1 — closed the `vh-agent-harness exec git <global-flag> <mutation>` commit-gate bypass.** A wrapped git mutation routed past a leading global flag — `vh-agent-harness exec git --no-pager commit`, `exec git -C /x push`, `exec git --git-dir=/x commit` — previously slipped past the commit-gate: the `git-mutation-bypass` adjacency regex (`\bgit\s+<mutation-verb>`) could not match a flag between `git` and the verb, and the harness branch returned `{action:"allow"}` after only the `isGateWrapperInDevShExec` check. The git-mutation parser (`walkGitGlobals`) never ran for wrapped payloads. v0.5.0 denies these forms at TWO layers (defense in depth): the A2 JS source-of-truth parser (`detectWrappedGitMutation` in `shell-guard-core.js`, reusing `walkGitGlobals` + `GIT_MUTATION_VERBS`) and the A1 Go binary backstop (`denyExecGitMutationPayload` in `internal/cli`, reusing `execro.GitVerbPastGlobals` + `permconfig.GitMutationVerbsSet`). Credit: TrueAI (responsible disclosure).
  - **exec-sandbox — new top-level command** `vh-agent-harness exec-sandbox <cmd>`. The authoritative OS-level enforcement layer behind the script-level exec-ro: pure-Go Landlock (filesystem integrity) + pure-Go seccomp-BPF (network/syscall hardening). Single static Go binary (`CGO_ENABLED=0`; new deps `github.com/landlock-lsm/go-landlock@v0.9.0` + `github.com/elastic/go-seccomp-bpf@v1.6.0`, both pure-Go — keeps the static-binary contract). Default **Profile B** (read repo root + system/tool paths; write only `./tmp/`; `.git` read-only; sibling repos denied). Network facet `--net=deny|allow|ask` (default deny). macOS is Linux-first (strict=fail-closed, best-effort=loud-warn+skip; no Seatbelt in v1). **HONESTY framing:** exec-sandbox is an **integrity + network** boundary, NOT a confidentiality/path-hiding boundary — denied paths stay VISIBLE (stat-able) but unwritable (EACCES on open-for-write).
  - **permconfig `tmp/**` allow** — the permission emitter now appends a `tmp/**: allow` override to every emitted **edit** block whose flat decision is not already broad-Allow, plus the top-level default block. Every agent's emitted `opencode.jsonc` gains repo-scoped scratch write on the edit/Write tool path on next `update`; all other edit decisions stay unchanged.
  - **Go 1.25.0 toolchain prerequisite** — `go.mod` now requires `go 1.25.0` (bumped from 1.22 for the new OS-primitive deps). Source-builders need Go 1.25+. Binary self-update (`vh-agent-harness update` from the installed binary) is unaffected; no breaking change to the shipped binary contract.

## What `update` handles for you
- Re-renders `.opencode/plugins/shell-guard-core.js` from the v0.5.0 template: adds `detectWrappedGitMutation(normalizedCmd, cwd)` and wires it into `evaluate()`'s harness branch before the allow. It tokenizes the wrapped payload, skips harness wrapper flags (`--service`/`--workdir`/`-w`/`--tty`/`-t`/`--` for `exec`; optional `--` for `exec-ro`), and if the payload starts with `git` calls the EXISTING `walkGitGlobals` to extract the verb past global flags; if the verb is in `GIT_MUTATION_VERBS` it returns a deny mirroring the `git-mutation-bypass` rationale. Reuses `walkGitGlobals` + `GIT_MUTATION_VERBS` — no new parser, no regex, no local verb copy.
- The Go binary backstop (`denyExecGitMutationPayload`) ships in the binary itself (not the rendered tree). It runs in `runExec` after the `pre_exec` hook and before `evaluateGate`, denying a bare git-mutation payload before the JS bridge is invoked.
- Re-emits every agent's `opencode.jsonc` `permission.edit` block with the `tmp/**: allow` carve-out appended last (findLast resolves tmp paths to allow); flat-allow agents (build, docs-steward) are unchanged because their broad allow already covers tmp.
- Ships the new `exec-sandbox` command surface in the binary (registered in `internal/cli/root.go`); `README.agent.md`'s command-surface entry is re-rendered from the core template.
- Bumps `go.mod` to `go 1.25.0` (source-build prerequisite; does not affect the rendered consumer corpus).
- No profile, ownership-class, capability, agent, skill, command-surface, or runtime change beyond the above. No schema-reconciliation change.
- Everything tagged `local_adopted` or inactive `overlay_extension` is preserved as before; a customized `shell-guard-core.js` triggers the standard courtesy check rather than being silently clobbered.

## Watch-outs
1. **The bypass was real and load-bearing.** Before v0.5.0, an agent (or operator) invoking `vh-agent-harness exec git --no-pager commit` (or any global-flag-prefixed mutation) bypassed the committer-exclusive commit-gate entirely — the mutation reached the backend with no gate. v0.5.0 closes it at two independent layers; either layer alone is sufficient. If your workflows relied on this form (they should not have — `AGENTS.md` routes all git writes through the committer agent / `commit-gate.sh`), switch to the blessed committer flow.
2. **Customized `shell-guard-core.js`.** If a previous `local_adopted` edit customized the plugin, the migration courtesy check fires and `update` will NOT silently clobber it. Review the diff and re-merge your customization onto the canonical fixed form so `detectWrappedGitMutation` lands.
3. **Nested shell strings are OUT OF SCOPE for F1.** `vh-agent-harness exec bash -c 'git commit …'` is NOT parsed by A1/A2's wrapped-mutation guard (neither parses nested shell strings). It remains governed by the EXISTING forbidden-pattern chain-guard scan (`bash -c` / `sh -c` / `eval` + git-mutation inspection), which already denies it. v0.5.0 changes nothing about that path.
4. **Read-only git and non-git exec are unaffected by F1.** `exec git --no-pager status`, `exec git -C /repo log`, `exec mkdir`, `exec pytest`, `exec npm test`, `exec bash -c 'echo hi'` all continue to pass through to the backend exactly as before. A1/A2 are scoped to git MUTATIONS only — they do not default-deny and do not call `execro.Classify`.
5. **exec-sandbox is Linux-first.** No Seatbelt profile in v1. On macOS, strict fails closed (no primitives); best-effort prints a loud warning and falls back to exec-ro classification. macOS must NEVER pretend protected. On Linux, the two-stage re-exec trampoline installs Landlock + seccomp in a hidden `__exec_sandbox_child` (new session/process group via `Setsid`, then `PR_SET_PDEATHSIG(SIGKILL)` → `SetNoNewPrivs` → seccomp `FlagTsync` → `landlock.V9.BestEffort().RestrictPaths` → `syscall.Exec`). Landlock is per-process/irreversible, so it MUST be in the child, not the parent.
6. **exec-sandbox HONESTY framing — integrity + network, NOT confidentiality/path-hiding.** Landlock is additive: denied paths stay VISIBLE (stat-able, metadata exposed) but unwritable (EACCES on open-for-write). The promise is "the command cannot WRITE or NETWORK outside the contract," NOT "the command cannot SEE anything." `stat ~/.ssh` succeeds (metadata visible); `ls ~/.ssh` fails (cannot open for listing); `touch ~/.ssh/foo` fails (EACCES). v1 accepts "inaccessible-but-visible" rather than adding bwrap path-hiding (deferred to roadmap). Do NOT rely on exec-sandbox to hide secrets from the sandboxed command.
7. **`--net=ask` cannot be auto-accepted by agents.** The ask prompt is TTY-interactive (operators only); the decision is resolved in the PARENT before forking the child. A non-TTY caller (agent) is hard-denied with a stderr notice and non-zero exit — agents CANNOT auto-accept network egress. Selective egress is deferred.
8. **`tmp/**` is an edit-tool carve-out only.** It widens the edit/Write tool path for repo-scoped `tmp/` scratch; it does NOT widen any bash/exec/path decision. tmp/ remains gitignored and watcher-ignored — the sanctioned disposable scratch surface, never committed, never watched.
9. **Go 1.25.0 source-build prerequisite.** `go.mod` requires `go 1.25.0`. Source-builders (CI, anyone rebuilding from the tag) need a Go 1.25+ toolchain. The shipped binary and binary self-update are unaffected — only building from source needs the newer toolchain.

## Rollback
A minor release is reversible by binary downgrade: reinstall the v0.4.1 binary and `vh-agent-harness update` re-renders the v0.4.1 corpus (the render is authoritative for `platform_managed` files). **Caveats:** rolling back (a) re-introduces the F1 bypass — `vh-agent-harness exec git --no-pager commit` would again reach the backend ungated; (b) removes the new `exec-sandbox` command surface; (c) reverts the `tmp/**` edit carve-out (read-only agents and the releaser regain a prompt when writing under tmp/); (d) reverts `go.mod` to `go 1.22`. Do not roll back if the F1 bypass is a live concern. No data migration ran, so there is nothing to undo beyond the binary reinstall and re-render. A `local_adopted` customization survives the rollback (it is preserved in both directions):
```bash
git checkout HEAD -- .opencode/         # only if an unrelated render diff is unwanted
```
