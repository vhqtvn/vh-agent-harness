# Migration: v0.2.0

## Summary
- **Release class:** minor (semver BREAKING). The Phase-5 capability-installer roster flip changes what `profile: minimal` renders, so this is NOT a patch.
- **Upgrade path:** binary self-update, then re-render the corpus. No manual file edits for the default case; consumers who relied on the Phase-3 full 20-agent roster under `profile: minimal` must choose a remediation (see How to migrate).
- **Risk:** medium. One breaking change (the roster flip, which also retires the Phase-3 bridge semantics for `coordination`/`web`) plus five additive/non-breaking changes.

## What changed (consumer-visible only)
| area | change | ships-via | class |
|------|--------|-----------|-------|
| Phase-5 roster flip | `profile: minimal` now resolves to the **8-agent baseline only** (was the full 20 under the Phase-3 backward-compat bridge). A profile declaring no explicit capabilities renders the 8 universal agents and NO cluster. The enum now means what it says. | binary (capability resolver) | **BREAKING** |
| `modules:` deprecation | The legacy `modules:` field is deprecated under the preset model. A non-empty `modules:` list surfaces a one-line warning on every `update`/`doctor`, nudging migration to `profile:` + `capabilities:`. Still parsed (existing profiles keep working); the values carry no effect. | binary (render warning) | non-breaking |
| new `capabilities:` field | The explicit opt-in UNION on top of the preset. `capabilities: [core/release]` adds that capability to whatever `profile:` selected — it adds, never replaces. | binary (capability resolver) | non-breaking |
| `profile:` enum semantics | `minimal`/`coordination`/`web` → empty preset (baseline-only = the 8 universal agents; `coordination`/`web` aliased to `minimal`, no cluster intrinsically scoped yet); `supervised` → `[core/gated-commit, core/debate]` (both clusters); unknown enum → empty (safe default). | binary (`profileCapabilityPresets`) | BREAKING (coordination/web no longer imply the bridge clusters) |
| `release` overlay pack | The FIRST embedded overlay pack ships the tag-driven `releaser` workflow. Selecting `core/release` (via `capabilities:`) OR listing `release` (via `overlays:`) renders the pack AND pulls `core/gated-commit` via hard-dep closure; the two selection paths converge. | binary (embedded overlay render) | non-breaking |
| embedded-default modules removal | `templates/core/.vh-agent-harness/vh-harness-profile.yml` no longer ships a `modules:` block, so fresh greenfield profiles never carry the deprecated field (and never trip the warning). | binary (embedded profile) | non-breaking |

## How to migrate (automated)
```bash
vh-agent-harness self-update            # pull the new binary (v0.2.0)
vh-agent-harness version                # expect: 0.2.0 (<label>)
vh-agent-harness update --dry-run       # ownership-safe preview; watch the roster + any modules: warning
vh-agent-harness update                 # applies platform_managed + active overlay_extension; resolves the new roster
vh-agent-harness doctor                 # lint the result
```

If you relied on the Phase-3 full 20-agent roster under `profile: minimal`, pick ONE remediation (the roster flip is the breaking change):
- set `profile: supervised` (restores both clusters: gated-commit + debate), OR
- keep `profile: minimal` and add `capabilities: [core/gated-commit, core/debate]` (explicit union).

Consumers who never depended on the bridge clusters (i.e. a `minimal` install was always intended to be lean) need do nothing.

## What `update` handles for you
- After `self-update`, the v0.2.0 binary resolves the roster from `preset(profile) ∪ explicit(capabilities:)` on every render (install, update, doctor's managed-drift re-render). No manual edit is required to adopt the model — `update` reads the live `vh-harness-profile.yml` and renders the gated agent blocks accordingly.
- A non-empty `modules:` in the live profile emits the one-line deprecation warning to stderr on `update` and `doctor` (count only; the values are not interpolated). The warning is advisory — `modules:` still parses, so nothing breaks.
- The `release` overlay pack renders into `.opencode/` on `update` only when selected (`capabilities: [core/release]` or `overlays: [release]`). It is opt-in; selecting it also materializes the `core/gated-commit` cluster via hard-dep closure.

## Watch-outs
1. **The Phase-5 flip is BREAKING.** Any consumer that relied on the Phase-3 full 20-agent roster under `profile: minimal` (or under `coordination`/`web`) loses the gated-commit and debate clusters after upgrade until it applies the remediation above. Audit the rendered `.opencode/` agents after the first v0.2.0 `update`.
2. **v0.1.9 binaries PREDATE project-overlay-pack rendering.** A consumer still on the released v0.1.9 binary cannot render project overlay packs, and its `update` may silently drop overlay edits. Upgrading to v0.2.0 is required to get overlay-pack rendering — do not attempt to consume overlay packs from a v0.1.9 binary.
3. **`modules: []` (empty list) after an armed reconcile.** The Go `Modules` field has no `omitempty`, so an armed reconcile emits `modules: []` (an empty list) rather than dropping the key. This is harmless — the deprecation warning fires ONLY on a NON-EMPTY `modules:`.

## Rollback
A minor/BREAKING release is reversible by binary downgrade: reinstall the v0.1.9 binary and `vh-agent-harness update` re-renders the Phase-3 corpus (the backward-compat bridge, which forces the full 20-agent roster regardless of `profile:`). The roster flip itself is therefore undoable. Caveats:
- `capabilities:` is forward-only — a v0.1.9 binary ignores it (it parses harmlessly). Your `capabilities:` list survives a rollback but does nothing until you re-upgrade.
- If you added a `release` overlay pack selection, the v0.1.9 binary cannot render it (see Watch-outs); revert the `overlays:`/`capabilities:` selection, or drop the rendered pack:
```bash
git checkout HEAD -- .opencode/         # drop an unwanted v0.2.0 render diff before downgrade
```
