# Migration: v0.16.0

## Summary
- **Release class:** minor (semver compatible). This release closes the §4.3
  defer-liveness fail-open (card `defer-003`): doctor check #12
  (`checkDeferLiveness`) previously SKIPped whenever
  `.local/coordinator/tasks/` was absent, which silently passed on a fresh
  checkout even for a repo that had already adopted the local coordinator
  transport. The fix introduces a committed, fresh-checkout-visible
  **coordinator-adoption marker** at `.vh-agent-harness/coordinator-adoption.json`
  (schema-v1, `{"version":1,"adopted":true}`). The gate can now distinguish
  "genuine greenfield, never adopted" (correct SKIP) from "adopted then whole
  transport deleted" (FAIL — the silent-pass-on-gitignored-`.local/` vector).
  The claims kernel only DERIVES marker state; the release gate is the sole
  transition authority that maps it to a check tier. This mirrors the
  `38c5c477` pattern (committed manifest curing a `.local/`-scan that silently
  passed). Last tag = `v0.15.1`; intended version = `v0.16.0`.
- **Upgrade path:** binary self-update, then re-render the corpus. The marker
  is NOT renderer-seeded on install — a greenfield install stays unadopted until
  its first coordinator-task write fires (create / update / ready / closeout /
  review / repair / activate — all route through `updateCoordinationTask`, the
  single write chokepoint), at which point the producer creates the marker
  idempotently. No profile, ownership-class, capability, or runtime change is
  required. `templates/core/` remains domain-free (token-only).
- **Risk:** low. The gate is fail-closed by design — it can only SKIP or refuse
  a release tag, never force one. Greenfield repos (no marker, no transport)
  keep the exact pre-v0.16.0 SKIP behavior. The only behavioral change is that a
  repo which had adopted the transport and then lost it now FAILs instead of
  silently passing. The existing card/CardError path is reached unchanged via
  the `tasks present + marker valid` row. The marker is unmanaged by the
  renderer (created at runtime, never seeded, never reconciled by `update`), so
  `vh-agent-harness update` cannot clobber a present marker.

## How to migrate (automated)
```bash
vh-agent-harness self-update            # pull the new binary (v0.16.0)
vh-agent-harness version                # expect: 0.16.0 (<label>)
vh-agent-harness update --dry-run       # ownership-safe preview
vh-agent-harness update                 # re-renders the state-lib producer
vh-agent-harness doctor                 # lint the result
```
No manual marker creation is required. The marker is created automatically on the
repo's next coordinator-task write (any create / update / ready / closeout /
review / repair / activate — all route through `updateCoordinationTask`, which
carries the producer). A greenfield install remains unadopted (SKIP) until then;
an adopted repo whose transport is intact stays green (valid marker + present
tasks → existing gate). Source-builders rebuilding from this tag are unaffected
(no Go toolchain bump).

## What `update` handles for you
- Re-renders `.opencode/scripts/state-lib.js` (template + rendered mirror) with
  the idempotent marker-creation block inside `updateCoordinationTask` (the
  single write chokepoint through which every coordinator-task op routes).
- Leaves any present `.vh-agent-harness/coordinator-adoption.json` untouched. The
  marker is unmanaged by the renderer (not in `lineage.yml` /
  `rendered-outputs.json`, not gitignored), so `update` neither seeds nor clobbers
  it — stronger than `project_owned`. `project_owned`
  (`internal/substrate/apply.go:268-274`) is present→preserve / absent→seed-from-
  template; it has NO never-seed variant. With no template it is inert yet
  misleading (claims a management contract that does not exist); with a template
  it would wrongly seed greenfield as adopted on first install. Unmanaged
  (template-less, never entering `walkStaged` / `planOutcome`) is the only
  ownership config that keeps a greenfield install unadopted while preserving a
  present marker across `update` — this is why the operator's `project_owned`
  instruction was reviewed and confirmed as a deviation from the correct choice.
- Leaves everything tagged `local_adopted`, `project_owned`, or inactive
  `overlay_extension` preserved as before. The default consumer's `opencode.jsonc`
  is byte-identical.

## Watch-outs
1. **The channel-liveness tradeoff (Option A).** The adoption marker closes
   WHOLE-DIR loss (the `.local/coordinator/tasks/` directory deleted or absent on
   a fresh checkout) but does NOT close selective single-card deletion — e.g. one
   errata card deleted while the directory remains. This is accepted because
   whole-dir loss is the structurally-forced gitignore vector (the entire
   `.local/` tree is gitignored transport), and an actor who can delete a single
   card can also delete the committed marker. Recording both halves here is
   deliberate: the marker is a liveness signal, not a card-by-card integrity
   guarantee.
2. **The marker is the opt-in, not `.local/`.** The gate deliberately does NOT
   infer adoption from the presence of `.local/coordinator/tasks/` (losable
   transport). Only the committed marker counts. A repo with transport present but
   no marker SKIPs (row 2) — this is intentional, to avoid promoting losable
   transport identity to canon.
3. **Greenfield stays green.** A fresh install (no marker, no transport) keeps the
   exact pre-v0.16.0 SKIP behavior. Do not pre-create the marker by hand — let the
   producer create it on the first real coordinator-task write, so the marker
   attests a real adoption rather than a manual file write.
4. **Corrupt marker FAILs.** A marker that is present but unparseable, or parses
   but does not satisfy `{version:1, adopted:true}`, produces `AdoptionMarkerCorrupt`
   → FAIL (fail-closed). Repair by deleting the corrupt marker (the next
   coordinator-task write via `updateCoordinationTask` recreates it) or fixing its contents.
5. **Authority line.** The claims kernel (`internal/memory/claims`) only DERIVES
   the marker state and projects it into the liveness result. It MUST NOT choose
   SKIP/FAIL/PASS. `release_gate.go` is the sole transition authority that maps
   the state to a tier. This mirrors how `validateCardIdentities` appends to
   `reg.CardErrors` without deciding the tier.

## Rollback
A minor release is reversible by binary downgrade: reinstall the `v0.15.1`
binary and `vh-agent-harness update` re-renders the `v0.15.1` corpus. **Caveats:**
rolling back (a) restores the unconditional `if !reg.TasksPresent { tierSkip }`
behavior in doctor check #12 — the §4.3 fail-open returns, but only affects a
repo that had adopted the transport and then lost it; (b) re-renders the older
  `.opencode/scripts/state-lib.js` without the producer block in
  `updateCoordinationTask` — a present marker
file is left in place (unmanaged) but no longer read by the gate; (c) does NOT
delete a committed `.vh-agent-harness/coordinator-adoption.json` — the file is
unmanaged and survives the rollback, but the v0.15.1 binary does not consult it.
No data migration ran, so there is nothing to undo beyond the binary reinstall
and re-render. A `local_adopted` customization survives the rollback as before.

## Non-consumer changes
This release is entirely consumer-facing release infrastructure (the §4.3
defer-liveness hardening is the product of this release). There are no
non-consumer commits in this arc.

`templates/core/` remains domain-free (token-only, no project literals) across
this entire arc. The canonical consumer source is the Go binary (the claims
kernel's `loadAdoptionMarker` + the release gate's state matrix) plus the
  embedded corpus re-rendered by `update` (the `updateCoordinationTask` producer
  block in `state-lib.js`, template + rendered mirror).
