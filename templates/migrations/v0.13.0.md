# Migration: v0.13.0

## Summary
- **Release class:** minor (semver compatible). The `v0.12.0..HEAD` arc (4 commits) introduces the **release-safety DEFER gate as the sole release-authority model**, with **two `feat(release):` commits** (3a7cd50, 38c5c47), **zero `!:` subjects**, and **no `BREAKING CHANGE:` footers** → MINOR bump; not patch/major. The arc retires the env-gated `legacy_fallback` release path (b9b3dba): manifest authority is now the ONLY release mode, and the `RELEASE_DEFER_MANIFEST_AUTHORITY` env switch is no longer consulted anywhere. Every consumer-visible change is release-infrastructure: the gate is fail-closed (a missing/malformed/stale committed manifest refuses the tag; override cannot cure evaluator-errors), the disposition matrix discloses conditional follow-ups, and the wrapper remains the sole pre-tag transition authority. Consumers not using the release overlay are unaffected — the commit-time promoter path (`mainPromoter`) is byte-identical. Last tag = `v0.12.0`; intended version = `v0.13.0`.
- **Upgrade path:** binary self-update, then re-render the corpus. Fully automatic for the default consumer. The release-mode evaluator (`mainRelease` in `check-defer-triggers.js`) and the manifest-aware releaser agent prompt ship in the Go binary + embedded corpus (acquired by `self-update`); the release-ceremony README section and the seed manifest re-render through `vh-agent-harness update`. No profile, ownership-class, capability, or runtime change is required for the default consumer. `templates/core/` remains domain-free (token-only).
- **Risk:** low. The release gate is fail-closed by design — it can only REFUSE a release tag, never force one. The committed manifest (`.vh-agent-harness/release-defer-dispositions.json`) is the sole release truth; on a fresh checkout it is visible and authoritative (closing the silent-pass-on-gitignored-`.local/` gap that motivated this arc). The override ceremony binds to the committed blob SHA (`git rev-parse HEAD:<path>`, not the worktree `hash-object`), so a dirty worktree cannot swap the bytes the ceremony confirms. Retiring `legacy_fallback` removes ~41 legacy-mode tests and the env dispatch router; the manifest-mode validation, handshake checks, disposition matrix, override ceremony, and CI recheck are preserved byte-identical. The promoter (commit-time DEFER check) is unchanged and stays non-blocking at commit time (hard non-goal).

## Watch-outs
1. **The release gate is fail-closed and manifest-authority-only.** There is no longer a `legacy_fallback` path. If the committed manifest at `.vh-agent-harness/release-defer-dispositions.json` is missing, malformed, stale (handshake SHAs do not match `HEAD^`), or carries an unresolved `override_required` record, the sanctioned wrapper REFUSES to create the tag. Override cannot cure evaluator-errors (missing/malformed/stale) — only repair of the committed manifest can.
2. **`RELEASE_DEFER_MANIFEST_AUTHORITY` is gone.** The env switch that previously gated between manifest-authority and legacy-fallback modes is no longer consulted anywhere (removed from the evaluator, the wrapper, the CI workflow, and the README). If your CI set `RELEASE_DEFER_MANIFEST_AUTHORITY=0` to force legacy mode, that setting is now silently ignored — remove it. The manifest path always runs.
3. **The manifest handshake is sacred and load-bearing.** The three SHA fields (`evaluated_commit`, `evaluated_tree`, `manifest_parent_commit`) must satisfy `evaluated_commit == HEAD^`, `evaluated_tree == tree(HEAD^)`, and `git diff HEAD^..HEAD == [manifest path only]` at tag time. The manifest-only commit M MUST be the final commit before tagging (never reordered before the note, never mixed with the note, never skipped). The releaser owns this ceremony end-to-end — operators do not recompute SHAs out-of-band.
4. **`.local/coordinator/tasks/` is provenance transport, not release truth.** The committed manifest is the sole release authority. Unpromoted `.local/` candidates may be lost — this is intentionally fine, because they are not trusted work yet. The release-mode evaluator reads ONLY the committed manifest (`HEAD:<path>`), never the gitignored `.local/` tree.
5. **The override ceremony binds to the committed blob SHA.** `--override-manifest-sha` must equal `git rev-parse HEAD:.vh-agent-harness/release-defer-dispositions.json` (the committed blob), NOT `git hash-object` of the worktree file. On a clean post-commit tree the two are identical, but binding to the committed blob prevents a dirty-worktree edit from swapping the bytes the ceremony confirms vs. the bytes CI verifies. The pre-`3e4ab4d` docs incorrectly suggested `git hash-object`; the production code was always correct.
6. **Two distinct failure classes.** The evaluator distinguishes (a) "release-relevant finding requires disposition" (classification=blocker, exit 1; remedy = resolve the finding OR supply the override ceremony) from (b) "manifest missing/malformed/stale" (classification=evaluator-error, exit 2; remedy = repair the committed manifest; override is inapplicable). Read the wrapper's refusal JSON `error` field to tell them apart.

## Rollback
A minor release is reversible by binary downgrade: reinstall the `v0.12.0` binary and `vh-agent-harness update` re-renders the `v0.12.0` corpus (the render is authoritative for `platform_managed` files). **Caveats:** rolling back (a) removes the release-safety DEFER gate's manifest-authority mode and restores the `legacy_fallback` path + the `RELEASE_DEFER_MANIFEST_AUTHORITY` env switch — consumers who relied on the manifest-authority model should re-add their committed manifest manually or stay on v0.13.0+; (b) re-renders the older `templates/core/.opencode/scripts/check-defer-triggers.js` (without the `mainRelease` manifest evaluator) — the commit-time promoter path is unaffected; (c) removes the manifest-aware `releaser` agent prompt and reverts to the pre-manifest-ceremony flow; (d) does NOT un-write a committed `.vh-agent-harness/release-defer-dispositions.json` — the file is seeded-once and survives the rollback, but the v0.12.0 binary does not consult it (the legacy path runs instead); (e) leaves `.vh-agent-harness/rendered-outputs.json` and `lineage.yml` semantics unchanged. No data migration ran, so there is nothing to undo beyond the binary reinstall and re-render. A `local_adopted` customization survives the rollback as before.
