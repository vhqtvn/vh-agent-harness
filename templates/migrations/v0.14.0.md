## Watch-outs
1. **`core/media-perception` is opt-in.** The capability-gated routing block renders to nothing when the capability is NOT selected â€” your existing caller prompts are unchanged unless you opt in. To enable, select the `core/media-perception` capability in your profile/capabilities union; then `vh-agent-harness update` re-renders the four caller agents with the routing guidance and the dual-channel handoff contract.
2. **`--net=ask` + `--sandbox=off` is now a hard error (D-7).** Previously this combination silently dropped the `ask` and ran un-sandboxed (fail-open). It is now rejected up front with a nonzero exit. If your scripts relied on the silent drop, switch to `--sandbox=best-effort` (or `strict`) so `ask` can actually be enforced, or set `--net=allow`/`--net=deny` explicitly.
3. **Strict NET decoding at the child boundary (D-5).** A missing/empty net policy now maps to `NetDeny` (not a silent pass), and `ask`/`unknown` decoded at the child boundary are now errors. Operators feeding a custom net policy to exec-sandbox children should ensure it decodes to `allow`/`deny`/`ask` cleanly.
4. **The shell-guard classifier is conservative, not a general parser.** Only single simple `rg`/`grep` commands using the closed flag whitelist `{ -F, --fixed-strings }` qualify for inert-literal treatment. `-Fi`, `-e`, `-A`, redirections, substitutions, pipelines, and any non-whitelisted node fail closed and the original deny stands. Use `vh-agent-harness exec-ro` (read-only, prompt-free) for legitimate search commands that the classifier cannot prove inert.
5. **The release-readiness boundary is fail-closed and dogfood-gated.** The deterministic G0â€“G7 gates and the exclusive `release-readiness-pass.json` artifact are release-time enforcement for this repo's dogfood. Consumers selecting the `release` overlay get the `releaser` ceremony (Step 3.2); the readiness REPORTER agent and the `scripts/release-tag.sh` wrapper are project-local dogfood infra that consumers author for themselves.
6. **Two distinct release-evaluator failure classes (unchanged from v0.13.0).** The DEFER manifest evaluator still distinguishes "release-relevant finding requires disposition" (classification=blocker, exit 1) from "manifest missing/malformed/stale" (classification=evaluator-error, exit 2). Override cannot cure evaluator-errors â€” only repair of the committed manifest can.

## Rollback
A minor release is reversible by binary downgrade: reinstall the `v0.13.1` binary and `vh-agent-harness update` re-renders the `v0.13.1` corpus (the render is authoritative for `platform_managed` files). **Caveats:** rolling back (a) removes the `core/media-perception` capability's caller routing â€” the four caller agents revert to their non-templated `.md` sources without the routing block; (b) reverts the shell-guard inert-literal classifier to the raw forbidden-pattern scan (legitimate `rg -F '/tmp' docs` will be denied again); (c) reverts the execsandbox D-4..D-8 hardening (restoring the `--sandbox=off` + `--net=ask` silent-drop and the lenient NET decoding); (d) reverts the scrubber consecutive-flag-leak fix (the bare-value dash-consumption regression returns); (e) removes the `releaser` Step 3.2 readiness ceremony; (f) does NOT un-write any committed manifest or readiness artifact â€” seeded files survive the rollback, but the v0.13.1 binary consults them per its own flow. No data migration ran, so there is nothing to undo beyond the binary reinstall and re-render. A `local_adopted` customization survives the rollback as before.
