# Migration: v0.9.0

## Summary
- **Release class:** minor (semver compatible). No breaking changes; no profile/ownership-class change. The `v0.8.1..HEAD` arc carries 3 `feat:` commits (`ecc06ad` auto-classifier-pilot embed, `10f511f` commit-review Spec axis, `ea1a4d3` predicted_impact + measured_outcome task-card fields), 1 `fix:` commit (`e122b45` commit-gate directory-deletion staging + path-staging contract), and 2 core-skill promotion commits (`c52268a` debugging-loop, `f56b964` tdd-loop), with **zero `!:` subjects and no `BREAKING CHANGE:` footers** → MINOR bump; not v1.0.0. Last tag = `v0.8.1`; intended version = `v0.9.0`.
- **Upgrade path:** binary self-update, then re-render the corpus. Fully automatic for the default consumer — every change ships through `go:embed` + `vh-agent-harness update`. The overlay pack renders only when a consumer selects it under `overlays:`; the two skills ship to every consumer but load only on use; the Spec axis and the two task-card fields are active on `update` (the Spec axis degrades to Standards-only when no task contract is bound, identical to pre-v0.9.0 behavior).
- **Risk:** low. The arc is additive end to end. The overlay pack is opt-in (a `minimal` profile that never names it renders nothing of it); the Spec axis degrades to Standards-only when no task contract is bound (identical to pre-v0.9.0 behavior); the two task-card fields are optional with no required migration; the two skills are additive files in `.opencode/skills/`.

The central thing to understand about this release: **the harness gains a second shipped embedded overlay pack, a dual-axis commit review, richer coordinator task cards, and two promoted core skills — all additive.** The headline for consumers is the `auto-classifier-pilot` embed (`ecc06ad`): the pack body is now genuinely distributed under `templates/overlays/`, so `overlays: [auto-classifier-pilot]` resolves and renders on the next `update`. Previously the name failed `open .: file does not exist` and blocked the whole render (the pack body was not distributed); the v0.7.0 "embedded and name-selectable" framing was aspirational on that point. v0.9.0 makes it real — `release` is no longer the only embedded pack. Alongside it, the commit-reviewer cascade gains a Spec axis (does the diff faithfully implement the originating task contract) evaluated alongside the existing Standards axis, the local task card gains two optional review fields, and two core skills (`debugging-loop`, `tdd-loop`) promote into the default-rendered skill set. A concurrent fix (`e122b45`) makes the shipped commit-gate stage whole deleted directories — previously a `fatal: not removing recursively without -r` abort — and documents the `--paths` path-staging contract in the gated-commit skill.

## Watch-outs
1. **The `auto-classifier-pilot` pack is OPT-IN and remains overlay-only.** It has no `capability-manifest.yml`, so it is selected solely via `overlays: [auto-classifier-pilot]` — never via `capabilities:`. A `minimal` profile that never names it renders nothing of it. A consumer who tried `overlays: [auto-classifier-pilot]` on a v0.7.0 or v0.8.x binary and hit `open .: file does not exist` will now find it resolves after upgrading to v0.9.0; no profile edit is needed beyond what they already had.
2. **The Spec axis is evaluated ONLY when a task contract is bound.** The commit-reviewer orchestrator calls `plan_state current_session`; if no session is bound (or `read_task_contract` returns nothing), the Spec axis is emitted `not_evaluated` with explicit disclosure and routing derives from the Standards axis ALONE — identical to the pre-v0.9.0 Standards-only review. Ad-hoc commits without a bound session are legitimate and are NOT fail-closed. The Spec axis never synthesizes a contract from the feature summary or the diff.
3. **`axis_conflict` is disclosure only, never a verdict.** It is emitted solely for a split-worthy divergence (the collapsed verdict is `split` because one axis approves while the other requests split). It is NOT emitted when the Spec axis was `not_evaluated`, when the verdict is `blocked`, or when both axes agree. The single top-level verdict stays `approve|blocked|split`.
4. **No cross-axis deduplication.** A Standards finding and a Spec finding may describe the same surface symptom while resting on different evidence; both are preserved, each tagged with its `axis`. Do not expect a one-to-one collapse between the two axes.
5. **The two new skills ship into `.opencode/skills/` for every consumer on the next `update`.** They are additive files; they load only when an agent invokes them. Each follows the S1 localization-split pattern: on first use the consumer authors a contracted `<repo>-debugging-loops.md` / `<repo>-tdd-seams.md` localization file (absence is step 0/1, not a failure) — the skill will not iterate on aspirational, unverified targets.
6. **`debugging-loop`'s D2C downgrade ENDS the agent-owned loop.** When deterministic red is genuinely unavailable, the skill declares a labeled downgrade, packages a handoff via `diagnostics-export`, and uses `bgshell-job` ONLY for long-running non-GPU shell probes — then stops. It does not simulate iteration on an unverifiable signal, and `bgshell-job` is explicitly NOT for GPU/thermal/compositor/visual-observation profiling.
7. **The two task-card fields are optional and lifecycle-gated.** `predicted_impact` is settable only in `draft`|`ready`; `measured_outcome` only in `reported`|`blocked`. Existing task cards deserialize unchanged (both default to null); there is no required migration and no data backfill.
8. **Directory-deletion staging is now supported; single-file behavior is unchanged.** A `--paths` entry naming a whole deleted directory acquires cleanly as of this release; previously it aborted the acquire with `fatal: not removing recursively without -r`. The `-r` flag is a no-op for single-file pathspecs, so consumers staging only literal file paths see no behavior change. `--paths` still accepts no globs — only literal file paths and directory pathspecs (see the gated-commit SKILL.md **Path staging** paragraph).

## Rollback
A minor release is reversible by binary downgrade: reinstall the `v0.8.1` binary and `vh-agent-harness update` re-renders the `v0.8.1` corpus (the render is authoritative for `platform_managed` files). **Caveats:** rolling back (a) removes the `auto-classifier-pilot` pack from the embedded set — the v0.8.1 binary predates its embed, so a consumer who selected it by name must either drop it from `overlays:` or keep a project-local pack at `.vh-agent-harness/overlays/auto-classifier-pilot/` (which still resolves via the project-first seam) before re-rendering with the older binary, otherwise the name fails to resolve again; (b) removes the Spec axis from the commit-reviewer cascade and `ship-review` (reviews revert to Standards-only, which is exactly the no-contract path, so gated commits behave identically); (c) drops the `predicted_impact` / `measured_outcome` fields from the rendered task commands/schema (existing task cards carrying the fields still deserialize — the fields are additive and ignored by the older reader); (d) removes the `debugging-loop` and `tdd-loop` skills from `.opencode/skills/`; (e) reverts the commit-gate directory-deletion staging fix (a `--paths` entry naming a deleted directory aborts again with `fatal: not removing recursively without -r`) and drops the Path-staging paragraph from the `gated-commit` SKILL.md. No data migration ran, so there is nothing to undo beyond the binary reinstall and re-render. A `local_adopted` customization survives the rollback and triggers the standard courtesy check on the way back up:
```bash
git checkout HEAD -- .opencode/         # only if an unrelated render diff is unwanted
```
