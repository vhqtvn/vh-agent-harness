# Erratum: v0.13.0 — release-defer-dispositions manifest ownership claim

> **Status:** staged for injection into the next release's migration note.
> When creating the next migration note, include a "Corrections to prior
> releases" section with the content below, then delete this file and
> dismiss the corresponding erratum card in `.local/coordinator/tasks/`.

## What was wrong

The v0.13.0 migration note (`templates/migrations/v0.13.0.md`) made two
false claims about the ownership and seeding of the
`.vh-agent-harness/release-defer-dispositions.json` manifest.

### 1. Ownership class — "seeded `platform_managed`" (line 12)

**Claimed (line 12, table row for commit `38c5c47`):** the manifest is
"seeded `platform_managed`" — i.e. the renderer seeds it from a template
and reconciles it on every `update`, the same lifecycle as the evaluator
script or the README.

**Actual behavior:** no template exists for
`.vh-agent-harness/release-defer-dispositions.json` (confirmed: zero matches
under `templates/`). The file is fully UNMANAGED by the renderer — never
seeded on install, never reconciled by `update`, never in `lineage.yml` /
`rendered-outputs.json`. It is runtime-committed by the releaser agent as
the single manifest-only commit M in the release ceremony. This is
structurally identical to the coordinator-adoption marker
(`.vh-agent-harness/coordinator-adoption.json`) introduced in v0.16.0:
git-tracked, unmanaged, runtime-created, never renderer-seeded.

### 2. Placeholder handshake SHAs (line 30)

**Claimed (line 30, "What `update` handles for you"):** "Seeds the
`release-defer-dispositions.json` manifest template for fresh installs of
the release overlay (schema v1; the seed carries placeholder handshake SHAs
that the releaser recomputes at each release-prep)."

**Actual behavior:** there is no seed/template at all. The live file
carries REAL handshake SHAs (e.g. `evaluated_commit` /
`manifest_parent_commit` = the actual commit the releaser evaluated;
`evaluated_tree` = the actual tree SHA). These are never placeholders —
they are recomputed by the releaser at each release-prep from the real
commit graph, then committed. A fresh install has NO manifest until the
releaser creates it during the first release ceremony.

## Impact

Low for release correctness. The release gate reads the committed manifest
via `HEAD:<path>` (committed blob) and validates the handshake SHAs against
the real commit graph regardless of how the file was created. The false
"seeded `platform_managed`" / "placeholder SHAs" claims describe a
non-existent template path; they do not affect the fail-closed gate
behavior or the override ceremony (which binds to the committed blob SHA).
The practical risk is operator confusion: an operator who believes a
template seeds the manifest may expect a fresh install to carry a manifest
and may not realize the releaser must create it from scratch on first
release. The correct mental model is "the releaser owns the manifest
end-to-end, from first creation through every recompute" — same as the
adoption marker's "the producer owns the marker from first write."
