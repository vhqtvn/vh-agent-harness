# Erratum: v0.12.0 — media-perception rendering claims

> **Status:** staged for injection into the next release's migration note.
> When creating the next migration note, include a "Corrections to prior
> releases" section with the content below, then delete this file and
> dismiss the corresponding erratum card in `.local/coordinator/tasks/`.

## What was wrong

The v0.12.0 migration note (`templates/migrations/v0.12.0.md`) made two
categories of false claim about the `core/media-perception` capability.

### 1. File rendering (lines 11, 28, 32)

**Claimed:** When `core/media-perception` IS selected, the agent prompt and
skill files additionally render. When NOT selected, none of these render.

**Actual behavior:** The agent prompt (`.opencode/agents/media-perception.md`)
and skill file (`.opencode/skills/media-perception/SKILL.md`) render on disk
as managed baseline by every `update`. The core renderer walks the whole
`templates/core/` source tree unconditionally, regardless of capability
selection. Only the `media-perception` agent registration block and the four
`media-perception: allow` caller edges in `opencode.jsonc` are capability-gated
via `{{- if .capabilities.media_perception }}`. When the capability is not
selected, those `opencode.jsonc` entries do not render and the agent is not
callable, but the prompt and skill files remain on disk.

Note: as of v0.15.0+, capability-owned CoreOutputs filtering was added so that
capability-owned files DO respect capability selection. The v0.12.0 claim was
false at the time of that release and remained false through v0.14.0.

### 2. YAML capability-list syntax (lines 11, 23, 32)

**Wrong:** `capabilities: [- core/media-perception]`

**Correct:** `capabilities: [core/media-perception]`

A leading `- ` inside an inline YAML sequence produces the wrong string value.
Use either the inline form `[core/media-perception]` or a multiline block:

```yaml
capabilities:
  - core/media-perception
```

## Impact

Low. The capability gate at the `opencode.jsonc` registration surface is
correct — the agent is not callable when unselected. The files-on-disk
behavior is cosmetic (inert prompt/skill present but unregistered).
