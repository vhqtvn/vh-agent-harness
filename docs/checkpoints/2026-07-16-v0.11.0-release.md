## The defect and the fix (substantive story)

During the release-readiness gate, a **harness-level ownership defect** surfaced:
the migration note `templates/migrations/v<next>.md` had **two conflicting
authors**. The shipped domain-free `releaser` authors it in Step 3 (Prepare),
while the dogfood-local `harness-release-readiness` G1 gate BLOCKED
release-readiness when the note was absent and delegated its authorship to
`docs-steward`. Two authoritative authors of one artifact is a contradiction →
deadlock.

A read-only `/solution-brief` (researcher → debate → planner) settled the
contract as **releaser-owned, cut-time authorship**: the shipped `releaser` is
the **sole semantic author** of the migration note; readiness G1 became a
read-only **authorability/lifecycle gate** with two PASS states
(`expected_at_cut`, `resumable_existing_note`) and a BLOCK state only when
version/coverage is ambiguous or the note conflicts. Absence alone is no longer
a blocker; inability to author without guessing still is. `docs-steward` retains
G3 ownership only.

Four contract edits landed in `1825d7e` (9 files, 593+/92−):

1. **`templates/overlays/release/agents/releaser.md`** (SOURCE, domain-free,
   ships to consumers) — sole-author declaration; Invariant 1a idempotency
   (≤1 note commit; a valid note already at HEAD → zero extra); Step 3 Prepare
   lifecycle table (`fresh` / `resumable_existing_note` / `refuse`); explicit
   Execute gate (note committed at HEAD before tagging); retry rule. Wrapper
   stays tag-only.
2. **`.vh-agent-harness/overlays/harness-dogfood/agents/harness-release-readiness.md`**
   (SOURCE, dogfood-local) — G1 → authorability gate; G5/G1 same-artifact rule
   rewritten; advisory `migration_note_coverage` JSON field added;
   `delegated_owners` G1 moved from `docs-steward` → `releaser`.
3. **`.vh-agent-harness/overlays/harness-dogfood/callable-graph-snippet.md`**
   (SOURCE) — readiness→docs-steward edge narrowed to G3 docs-coverage
   remediation only; readiness→releaser handoff may carry
   `expected_at_cut` / `resumable_existing_note`.
4. **`internal/cli/release_note_ownership_test.go`** (NEW regression test,
   pkg `cli`) — pins sole-author + G1-off-`docs-steward` + idempotency in both
   authoritative sources and rendered mirrors.

Regenerated mirrors: `.opencode/agents/releaser.md`,
`.opencode/agents/harness-release-readiness.md`,
`.opencode/docs/agents/callable-graph.md`, `.vh-agent-harness/lineage.yml`,

## Contradictions

- **G1 ↔ releaser authorship deadlock (THE release's contradiction — now resolved).**
  The migration note had two authoritative authors; this was the contradiction
  the release exists to resolve. `1825d7e` settled it: `releaser` is the sole
  semantic author, G1 is a read-only authorability gate. No longer live.
- **Stale push-status assumption (resolved this session).** The release brief
  that authorized this checkpoint stated the push was "OPERATOR-PENDING" and
  instructed not to claim the tag was pushed. Live `git ls-remote --tags origin
  v0.11.0` returns the tag (`08c5984`) and `git ls-remote origin HEAD` returns
  the released commit (`7f29378`) — so the operator ran the push after the brief
  was written. This record reflects the verified pushed state rather than the
  stale pending state. No blocking impact; the release is complete.
- **SHA-precision nuance on `v0.10.0` (not a material contradiction).** The brief
  cited `v0.10.0` parenthetically as "commit `8b64eaf`"; `8b64eaf` is the tag
  **object**, which dereferences to commit `706961c`. The arc and bump analysis
  are unaffected. Recorded with the corrected SHAs above.
